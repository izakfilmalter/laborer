## 2026-03-09: Issue 4 — SidecarManager Rust module — spawn and kill processes

Status: COMPLETED

### What was done
- Created `apps/web/src-tauri/src/sidecar.rs` — a new Rust module containing:
  - **Shell environment probing**: Probes the user's login shell (`$SHELL` or `/bin/zsh`) using
    `env -0` (NUL-delimited output) to capture PATH and other variables that macOS GUI apps
    don't inherit. Tries interactive-login (`-il`) first, falls back to login-only (`-l`).
    Skips nushell (unsupported). 5-second timeout prevents hanging on slow shell configs.
  - **`SidecarManager` struct**: Stored in Tauri managed state. Probes shell env once at
    construction. Tracks spawned sidecars via `Mutex<Vec<TrackedSidecar>>`.
  - **`spawn()`**: Resolves sidecar binary path relative to `tauri::process::current_binary()`.
    Uses `tokio::process::Command` with `process-wrap` `ProcessGroup::leader()` on Unix for
    clean group-kill. Passes PORT, TERMINAL_PORT, DATA_DIR env vars merged with shell env.
    Pipes stdout/stderr to `log::info!`/`log::warn!` via async tokio tasks.
  - **Process lifecycle monitoring**: Async tokio task polls `child.try_wait()` and listens for
    kill signals via `mpsc::channel`. Emits `TerminatedPayload` (code + signal) via oneshot
    channel on exit.
  - **`kill_all()`**: Sends kill signal to all tracked sidecars and waits 500ms for cleanup.
  - **`take_exit_rx()`**: Allows taking the exit receiver for health check racing (used by Issue 5).
- Updated `apps/web/src-tauri/src/lib.rs`:
  - Added `mod sidecar;` declaration.
  - Creates `SidecarManager` (via `Arc`) before building the Tauri app.
  - Registers it as Tauri managed state via `.manage()`.
  - Changed `.run()` to `.build()` + `app.run()` to capture `RunEvent::Exit`.
  - On `RunEvent::Exit`, spawns a cleanup thread with a new tokio runtime to `kill_all()`.
- Updated `Cargo.toml` with new dependencies:
  - `tokio = { version = "1", features = ["process", "io-util", "time", "sync", "rt"] }`
  - `process-wrap = { version = "9", features = ["tokio1"] }`

## 2026-03-09: Issue 5 — SidecarManager — health checks and Tauri events

Status: COMPLETED

### What was done
- Extended `apps/web/src-tauri/src/sidecar.rs` with health check, state tracking, crash
  detection, Tauri event emission, and restart support:
  - **SidecarState enum**: `Starting`, `Healthy`, `Crashed { error }`, `Stopped` — serializable
    as JSON with `#[serde(rename_all = "snake_case", tag = "status")]`.
  - **Health check polling** (`check_health`): Uses `reqwest` with a 2-second timeout, no-proxy
    for localhost. Polls at 100ms intervals after sidecar spawn.
  - **`spawn_and_wait_healthy()`**: New high-level method that spawns a sidecar and races three
    futures: health poll (100ms interval), process termination detection, and a 10-second timeout.
    Emits `sidecar:healthy` or `sidecar:error` Tauri events based on the outcome.
  - **Tauri event payloads**: `SidecarHealthyPayload { name }` for `sidecar:healthy`,
    `SidecarErrorPayload { name, error, last_stderr }` for `sidecar:error`.
  - **Stderr buffering**: Each sidecar maintains a ring buffer of the last 50 stderr lines
    (via `Arc<Mutex<Vec<String>>>`), included in error payloads for crash diagnostics.
  - **Crash detection** (`start_crash_monitor`): After a sidecar becomes healthy, a background
    tokio task monitors the exit receiver. If the process exits while in the `Healthy` state,
    it transitions to `Crashed` and emits `sidecar:error`.
  - **`kill_one()`**: Kills a specific sidecar by name, sets state to `Stopped`, waits 200ms.
  - **`restart()`**: Kills (if running) and re-spawns with full health check cycle.
  - **`restart_sidecar` Tauri command**: Exposed as `#[tauri::command]` for frontend invocation,
    takes a `SidecarName` parameter and delegates to `SidecarManager::restart()`.
  - **`get_state()`**: Public method to query current sidecar state.
- Updated `apps/web/src-tauri/src/lib.rs`:
  - Registered `restart_sidecar` in `invoke_handler`.
- Updated `Cargo.toml`:
  - Added `reqwest = { version = "0.12", default-features = false, features = ["rustls-tls"] }`.
- 15 unit tests covering: all existing tests + state serialization, payload serialization,
  health URL generation, health check failure for nonexistent server, state equality,

## 2026-03-09: Issue 2 — Compile server service as standalone binary

Status: COMPLETED

### What was done
- Rewrote `apps/web/scripts/build-sidecars.ts` to support the server sidecar compilation:
  - The server sidecar now uses `Bun.build()` JavaScript API instead of the CLI to enable
    build plugins for handling native dependencies.
  - Added `BuildPlugin` interface and inline Bun type declarations (the web app tsconfig
    doesn't include Bun types).
  - Removed the `installExternalPackages()` function and `node_modules`-based approach.
    All dependencies are now bundled directly into the binary.
  - Added `copyFiles` config to sidecar definitions for shipping companion files alongside
    the binary (wa-sqlite.node.wasm).

- **Build plugins resolve three native dependency issues:**
  1. **`@parcel/watcher` stub**: `@effect/platform-node-shared` statically imports `@parcel/watcher`
     which crashes compiled binaries because the native `.node` addon can't be loaded from
     Bun's virtual `/$bunfs/root/` filesystem. The plugin stubs it with an empty module.
     The server's own `file-watcher.ts` gracefully falls back to `fs.watch`.
  2. **`@effect/platform-node/Undici.js` stub**: This module re-exports `undici` via
     `export * from "undici"`, but Bun's bundler generates a `__reExport(exports, undici)`
     call referencing an undeclared `undici` namespace variable (bundler bug). Since the
     server uses `@effect/platform-bun` (not node), this module is never used at runtime.
  3. **wa-sqlite WASM path patch**: The Emscripten-generated `wa-sqlite.node.mjs` loader
     resolves `.wasm` files via `new URL("wa-sqlite.node.wasm", import.meta.url).href`.
     In compiled binaries, `import.meta.url` points to `/$bunfs/root/...`. The plugin
     patches this to resolve relative to `process.execPath` (the real binary on disk).

- **wa-sqlite.node.wasm**: The 604KB WASM file is copied alongside the compiled binary
  via the `copyFiles` config. LiveStore's SQLite adapter loads it at runtime for
  server-side persistence.

- **mcp-registrar.ts path resolution**: Already handled in a prior commit — uses
  `IS_COMPILED_BINARY` flag to detect compiled mode and resolves `laborer-mcp` as a
  sibling of `process.execPath` instead of using `import.meta.url`.

- **Decision: Bundle all JS deps, externalize only `lightningcss`**. The original plan
  was to mark `@livestore/adapter-node` and `@parcel/watcher` as `--external` and install
  them in a `node_modules/` directory alongside the binary. This approach failed because
  Bun compiled binaries cannot resolve external `import()` calls — they resolve from the
  virtual `/$bunfs/root/` path, not the binary's real directory. The solution is to bundle
  everything directly and handle native deps via build plugins (stubs + patches).

## 2026-03-09: Issue 7 — Frontend crash notification and restart UI

Status: COMPLETED

### What was done
- Created `apps/web/src/hooks/use-sidecar-crash-listener.ts` — a React hook that:
  - **Tauri-only activation**: Checks `isTauri()` before registering any listeners. In
    browser/non-Tauri mode, the hook is a complete no-op.
  - **Dynamic imports**: Uses `await import('@tauri-apps/api/event')` and
    `await import('@tauri-apps/api/core')` to avoid bundling Tauri APIs in non-Tauri builds.
  - **`sidecar:error` listener**: Listens for crash events with payload
    `{ name: "server"|"terminal", error: string, last_stderr: string }`.
    Shows a persistent `toast.error()` with:
    - Service display name (e.g., "Server service crashed")
    - Error description from the payload
    - `duration: Infinity` — toast stays until dismissed or service recovers
    - "Restart" action button that invokes `restart_sidecar` Tauri command
  - **`sidecar:healthy` listener**: Listens for recovery events with payload
    `{ name: "server"|"terminal" }`. Dismisses the error toast for the recovered
    service and shows a success toast ("Server service recovered").
  - **Restart flow**: On "Restart" click, dismisses the error toast, shows a loading
    toast ("Restarting Server..."), invokes `restart_sidecar` Tauri command. On failure,
    dismisses the loading toast and shows a new error toast with the failure reason.
  - **Toast ID tracking**: Uses a `Map<SidecarName, string|number>` ref to track active
    error toast IDs per service, enabling targeted dismissal. If a new error arrives for
    a service that already has an error toast, the old one is dismissed first.
  - **Cleanup**: Returns an effect cleanup function that unregisters all Tauri event
    listeners and sets a `cancelled` flag to prevent late registration from async setup.

## 2026-03-09: Issue 8 — MCP symlink creation on app launch

Status: COMPLETED

### What was done

#### Symlink creation (`sidecar.rs`)
- Added `MCP_SYMLINK_PATH` constant (`/usr/local/bin/laborer-mcp`).
- Added `SidecarManager::mcp_binary_path()`: Resolves the path to the `laborer-mcp`
  binary inside the app bundle, as a sibling of the main Tauri binary (via
  `tauri::process::current_binary()`).
- Added `SidecarManager::create_mcp_symlink()`: Creates or updates the symlink at
  `/usr/local/bin/laborer-mcp` pointing to the MCP binary inside the app bundle:
  - If the symlink already exists and points to the correct target, skips (no-op).
  - If the symlink exists but points to a different target (old version), removes
    and recreates it.
  - If a non-symlink regular file exists at the path, skips to avoid overwriting.
  - If the MCP binary is not found in the bundle, logs a warning and skips.
  - If symlink creation fails (e.g., permission denied on `/usr/local/bin/`),
    logs a warning with a manual `sudo ln -sf` command but does NOT block app startup.
  - Unix-only implementation (`std::os::unix::fs::symlink`); logs a warning on
    non-Unix platforms.

#### LABORER_MCP_PATH env var (`sidecar.rs`)
- Updated `SidecarManager::sidecar_env()` to accept `app: &tauri::AppHandle`
  (was previously `&self, name` only) so it can resolve the MCP binary path.
- When building env vars for the **server** sidecar, adds `LABORER_MCP_PATH`:
  - If the symlink at `/usr/local/bin/laborer-mcp` exists, passes the symlink
    path so AI agent configs use the well-known stable path.
  - If the symlink doesn't exist (e.g., permission failure), falls back to the
    MCP binary path inside the app bundle.

#### App setup orchestration (`lib.rs`)
- In release mode, calls `SidecarManager::create_mcp_symlink()` BEFORE spawning
