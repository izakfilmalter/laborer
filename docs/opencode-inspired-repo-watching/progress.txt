## Issue 3: Scoped repository watcher coordinator — COMPLETED

### What was done
- Created `FileWatcher` abstracted watcher backend (`packages/server/src/services/file-watcher.ts`)
  - Platform-independent interface wrapping `fs.watch` behind a tagged Effect service
  - Exposes `subscribe(path, onChange, onError, options)` returning a `WatchSubscription` handle
  - Designed to be swappable with `@parcel/watcher` or other native backends
  - Default `Layer.succeed` implementation using Node.js `fs.watch`

- Created `RepositoryWatchCoordinator` scoped Effect service
  (`packages/server/src/services/repository-watch-coordinator.ts`)
  - Owns all watcher lifecycle for each registered repository
  - Subscribes to canonical common git directory for metadata changes
  - Uses `RepositoryIdentity` for canonical path resolution (not raw paths)
  - Debounces filesystem events (500ms) before triggering reconciliation
  - Two-phase git dir watching: watches `.git` initially, switches to
    `.git/worktrees` when the subdirectory appears
  - `watchProject(projectId, repoPath)` — idempotent, replaces previous watchers
  - `unwatchProject(projectId)` — tears down subscriptions and clears timers
  - `watchAll()` — reconciles and watches all persisted projects (startup bootstrap)
  - `Effect.addFinalizer` ensures all watchers close on scope disposal

- Replaced `WorktreeWatcher` with `RepositoryWatchCoordinator` throughout:
  - `ProjectRegistry` now depends on `RepositoryWatchCoordinator` instead of `WorktreeWatcher`
  - `main.ts` layer graph updated: `RepositoryWatchCoordinator.layer` + `FileWatcher.layer`
    replace `WorktreeWatcher.layer`
  - All test layers updated (worktree-watcher.test.ts, repository-identity.test.ts,
    project-registry.worktree-detection.test.ts, workspace-destroy-origin.test.ts,
    rpc/test-layer.ts)

- Added 5 new integration tests (`packages/server/test/repository-watch-coordinator.test.ts`)
  - Each registered project gets a scoped watcher coordinator tied to its lifecycle
  - Coordinator watches the canonical common git dir for metadata changes
  - Removing a project tears down its watchers cleanly (no detection after unwatch)
  - Server shutdown tears down all watcher resources through scoped service disposal
    (verified via recording FileWatcher mock)
  - FileWatcher abstraction is used for subscriptions instead of raw fs.watch
    (verified via recording FileWatcher mock)

- Preserved all existing worktree-watcher behavior (4 tests renamed but unchanged):
  - Reconciles on worktree add and remove
  - unwatchProject stops future reconciliation
  - watchAll reconciles existing projects and starts watchers
  - Handles repos with no .git/worktrees until first linked worktree

- All 145 tests pass (24 test files)
- TypeScript compiles with no errors

## Issue 4: Git metadata refresh for branch and worktree changes — COMPLETED

### What was done
- Added `workspaceBranchChanged` event to LiveStore schema (`packages/shared/src/schema.ts`)
  - New synced event `v1.WorkspaceBranchChanged` with `id` and `branchName` fields
  - Materializer updates the `branchName` column on the `workspaces` table
  - Enables updating branch names on existing workspace records (previously only
    set at creation time and never updated)

- Created `BranchStateTracker` Effect service (`packages/server/src/services/branch-state-tracker.ts`)
  - Reads current branch for each workspace via `git rev-parse --abbrev-ref HEAD`
  - Compares git-reported branch with stored `branchName` in the workspace record
  - Commits `workspaceBranchChanged` events for any stale branch names
  - Handles detached HEAD state (formats as `detached/<short-sha>`)
  - Skips destroyed workspaces during refresh
  - Returns `BranchRefreshResult` with `checked` and `updated` counts

- Updated `RepositoryWatchCoordinator` to integrate branch refresh
  (`packages/server/src/services/repository-watch-coordinator.ts`)
  - Added `BranchStateTracker` as a service dependency
  - Event classification: git dir events are now classified by concern using
    `isBranchRelatedEvent()` and `isWorktreeRelatedEvent()` helper functions
  - Branch-related files: HEAD, MERGE_HEAD, REBASE_HEAD, ORIG_HEAD, FETCH_HEAD,
    and anything under `refs/` trigger branch refresh
  - Worktree-related files: `worktrees/` directory changes trigger reconciliation
  - Per-concern debouncing: separate `worktreeTimer` and `branchTimer` on each
    `ProjectWatcherState` so branch refresh and worktree reconciliation debounce
    independently (both at 500ms)
  - `watchAll()` now runs both initial reconciliation AND initial branch refresh
    at startup
  - Watcher error handlers schedule both reconcile and branch refresh

- Wired `BranchStateTracker.layer` into the layer graph across the codebase:
  - `main.ts`: added between `RepositoryWatchCoordinator.layer` and `FileWatcher.layer`
  - All test layers updated with import and layer provision:
    - `repository-watch-coordinator.test.ts` (main TestLayer + 2 inline ScopedTestLayers)
    - `worktree-watcher.test.ts`
    - `project-registry.worktree-detection.test.ts`
    - `workspace-destroy-origin.test.ts`
    - `repository-identity.test.ts`
    - `rpc/test-layer.ts` (both RPC test layer variants)

- Added 7 new integration tests (`packages/server/test/branch-state-tracker.test.ts`)
  - BranchStateTracker unit tests (5):
    - Refreshes branch name when workspace branch is stale
    - Does not update when branch name is already current
    - Refreshes multiple workspaces in one pass
    - Skips destroyed workspaces during branch refresh
    - Detects detached HEAD state during branch refresh
  - Coordinator integration tests (2):
    - Branch switch on main worktree triggers branch refresh through the coordinator
    - Worktree metadata changes trigger both reconciliation and branch refresh

## Issue 5: Repo-wide file events with normalized event bus and ignores — COMPLETED

### What was done
- Created `RepositoryEventBus` Effect service (`packages/server/src/services/repository-event-bus.ts`)
  - Publishes normalized add/change/delete file events for repository watching
  - Stable `RepositoryFileEvent` shape: `{ type, relativePath, absolutePath, projectId, repoRoot }`
  - Multiple downstream subscribers can consume events without creating duplicate watchers
  - `subscribe(handler)` returns an `EventBusSubscription` with synchronous `unsubscribe()`
  - `publish(event)` fans out to all registered handlers
  - `normalizeEvent(params)` applies ignore rules and returns null for suppressed paths

- Added centralized ignore handling with `shouldIgnore` and `DEFAULT_IGNORED_PREFIXES`
  - Prefix-based matching against the first path segment of relative paths
  - Default ignores: `.git`, `node_modules`, `dist`, `build`, `out`, `.next`, `.nuxt`,
    `.svelte-kit`, `.turbo`, `.yarn`, `.pnpm-store`, `.idea`, `.vscode`, `.DS_Store`,
    `Thumbs.db`, `coverage`, `.nyc_output`
  - Both `shouldIgnore` and `DEFAULT_IGNORED_PREFIXES` are exported for testability

- Updated `RepositoryWatchCoordinator` to wire repo-wide file watching
  (`packages/server/src/services/repository-watch-coordinator.ts`)
  - Added `RepositoryEventBus` as a service dependency
  - Added `repoRootSubscription` to `ProjectWatcherState` for repo checkout root watching
  - New `subscribeRepoRoot(projectId, repoPath)` subscribes to the checkout root
    with `recursive: true` via `FileWatcher`
  - Raw watcher events are mapped through `eventBus.normalizeEvent()` — ignored paths
    are suppressed before publishing, non-ignored events are published via `eventBus.publish()`
  - `watchProject` now subscribes to both the git dir (metadata) AND the repo root (files)
  - `closeState` tears down both git dir and repo root subscriptions on project removal
  - Updated JSDoc header to reflect Issue 5 scope (no longer "future")

- Wired `RepositoryEventBus.layer` into the layer graph across the codebase:
  - `main.ts`: added between `BranchStateTracker.layer` and `FileWatcher.layer`
  - All test layers updated with import and layer provision:

    - `branch-state-tracker.test.ts`

    - `repository-identity.test.ts`
    - `rpc/test-layer.ts` (both RPC test layer variants)

- Added 18 new tests (`packages/server/test/repository-event-bus.test.ts`)
  - shouldIgnore unit tests (7):
    - Ignores .git paths
    - Ignores node_modules paths
    - Ignores dist, build, and out paths
    - Ignores framework-specific output directories (.next, .nuxt, .svelte-kit, .turbo)
