## Session: Issue #2 — PtyHostClient Effect service

### Task selected
Issue #2: PtyHostClient Effect service — spawn and route events.
Priority rationale: Issue #1 (PTY Host script) is already complete. Issue #2 is the critical-path blocker — it unblocks Issues #3, #4, #5, #6, #7, and #8. Without PtyHostClient, no other issue can proceed.

### What was built
Created `packages/server/src/services/pty-host-client.ts` — a new Effect service that:

1. **Spawns the PTY Host** as a Bun child process during layer construction (`Bun.spawn` with stdin/stdout pipes).
2. **Waits for the `ready` event** using an Effect `Deferred` before the layer is considered constructed.
3. **Provides typed methods**: `spawn()`, `write()`, `resize()`, `kill()` — each sends a JSON command to the PTY Host via stdin.
4. **Implements a line-based JSON parser** on stdout via an async `ReadableStream` reader with a line buffer.
5. **Routes `data` events** to per-terminal data callbacks (registered during `spawn()`).
6. **Routes `exit` events** to per-terminal exit callbacks and cleans up callback maps.
7. **Monitors the PTY Host process** for crashes and invokes registered crash callbacks.
8. **Registers a finalizer** that kills the PTY Host child process on layer teardown.

Also wired `PtyHostClient.layer` into `main.ts` layer graph alongside `TerminalManager.layer`.

## Session: Issue #3 — TerminalManager refactor to use PtyHostClient

### Task selected
Issue #3: TerminalManager refactor — use PtyHostClient instead of node-pty.
Priority rationale: Issue #3 is the critical-path blocker — it unblocks Issues #4, #5, and #8. It is the core integration that makes the terminal feature actually work end-to-end through the PTY Host architecture. Without this, terminals still fail due to the SIGHUP issue in the main server process.

### What was built
Refactored `packages/server/src/services/terminal-manager.ts` to delegate all PTY operations to `PtyHostClient` instead of importing and calling node-pty directly.

Key changes:

1. **Removed node-pty dependency**: Removed `import("node-pty")` lazy import and `type { IPty }` import. TerminalManager no longer references node-pty in any way.
2. **Added PtyHostClient dependency**: TerminalManager now `yield* PtyHostClient` during layer construction, accessing the IPC client for all PTY operations.
3. **Simplified ManagedTerminal interface**: Removed the `pty: IPty` field. ManagedTerminal now only holds metadata (`id`, `workspaceId`, `command`), since the actual PTY instance lives in the PTY Host child process.
4. **Refactored spawn()**: Uses `ptyHostClient.spawn()` with data and exit callbacks instead of `nodePty.spawn()`. Data callback decodes base64 from the PTY Host before committing `terminalOutput` events to LiveStore. Exit callback updates LiveStore status and cleans up in-memory state, same as before.
5. **Refactored write()**: Delegates to `ptyHostClient.write(id, data)` instead of `terminal.pty.write(data)`.
6. **Refactored resize()**: Delegates to `ptyHostClient.resize(id, cols, rows)` instead of `terminal.pty.resize(cols, rows)`.
7. **Refactored kill()**: Delegates to `ptyHostClient.kill(id)` instead of `terminal.pty.kill()`.
8. **Refactored killAllForWorkspace()**: Uses `ptyHostClient.kill()` for each terminal instead of `terminal.pty.kill()`.
9. **Removed debug PTY spawn block**: Deleted the entire `require("node-pty")` debug comparison spawn (lines 259-287 of original) which was a debugging artifact.
10. **Base64 decoding**: Added `Buffer.from(base64Data, "base64").toString("utf-8")` in the data callback to decode PTY Host output before committing to LiveStore, since xterm.js expects raw terminal output strings.
11. **ptySessionRef**: Changed from `String(pty.pid)` (which required a local IPty reference) to the terminal `id` string, since the PID is only known inside the PTY Host process.

### Files changed
- `packages/server/src/services/terminal-manager.ts` — REFACTORED (537 → ~460 lines)

### Public interface
The public interface of `TerminalManager` is **unchanged** — all existing RPC handlers (`terminal.spawn`, `terminal.write`, `terminal.resize`, `terminal.kill`, `rlph.*`) continue working without modification.

## Session: Issue #4 — Stale terminal cleanup on server startup

### Task selected
Issue #4: Stale terminal cleanup on server startup.
Priority rationale: Issue #4 is a correctness requirement that directly impacts user experience. Without it, server crashes leave ghost "running" terminals in LiveStore that confuse users. It also handles PTY Host crashes gracefully (marking all terminals as stopped). This is the most impactful remaining task for end-to-end reliability.

### What was built
Added two pieces of cleanup logic to the `TerminalManager` layer construction in `packages/server/src/services/terminal-manager.ts`:

1. **Stale terminal cleanup on startup**: During layer construction (before the service starts accepting requests), queries LiveStore for all terminals with status `"running"` and commits `v1.TerminalStatusChanged { status: "stopped" }` for each. These are orphans from a previous server crash — the PTY processes are long gone. Logs the count of cleaned-up terminals via `Effect.log` for observability.

2. **PTY Host crash handler**: Registers an `onCrash` callback with `PtyHostClient`. When the PTY Host process exits unexpectedly, the callback:
   - Reads all terminal IDs from the in-memory `terminalsRef` map
   - Commits `v1.TerminalStatusChanged { status: "stopped" }` for each
   - Clears the in-memory map (all terminals are dead)
   - Logs the event via `Effect.log` for observability

Both use `Effect.log` with `Effect.annotateLogs("module", "TerminalManager")` for structured logging. The crash handler uses `Runtime.runSync` (already extracted during layer construction) since it runs in a plain JS callback context.
