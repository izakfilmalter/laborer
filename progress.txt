## Issue 7: LiveStore schema — PanelLayout table + events + materializer — COMPLETED

Date: 2026-03-04

### What was done
- Defined `panelLayout` table using `State.SQLite.table` with columns: `id` (text, primary key), `layoutTree` (json, backed by `PanelNodeSchema` from types.ts for automatic JSON serialization/deserialization), `activePaneId` (text, nullable).
- Used `State.SQLite.json({ schema: PanelNodeSchema })` which leverages LiveStore's first-class JSON column support — the underlying SQLite column is text, but `Schema.parseJson(PanelNodeSchema)` is applied automatically for encode/decode. This means the recursive `PanelNode` tree (with `LeafNode` and `SplitNode` variants) is stored as JSON text and validated on read.
- Defined 4 synced events using `Events.synced` with a shared `layoutEventSchema` containing `{ id, layoutTree: PanelNodeSchema, activePaneId: NullOr(String) }`:
  - `layoutSplit` (name: `v1.LayoutSplit`) — emitted when a pane is split
  - `layoutPaneClosed` (name: `v1.LayoutPaneClosed`) — emitted when a pane is closed
  - `layoutPaneAssigned` (name: `v1.LayoutPaneAssigned`) — emitted when pane content changes
  - `layoutRestored` (name: `v1.LayoutRestored`) — emitted when full layout is set/restored
- All 4 events carry the complete updated layout tree (not diffs). Tree manipulation logic will live in the React app; materializers simply persist the result. Each event type exists for auditability of user actions.
- Defined materializers for all 4 events using `panelLayout.insert({ id, layoutTree, activePaneId }).onConflict("id", "replace")` — upsert pattern ensures a single row per session is maintained.
- Added `panelLayout` to the `tables` export and all 4 events to the `events` export.
- Updated `state` and `schema` exports to include the new table and events.
- Imported `PanelNodeSchema` from `./types.js` — the `Schema` from `@livestore/livestore` is a re-export of `effect`'s `Schema`, so the types are fully compatible.
- Verified: `tsc --noEmit` passes (no type errors — only pre-existing warnings from Effect language service plugin about duplicate `@effect/platform` versions), `biome check` passes on schema.ts (no lint issues), server package type-checks cleanly.

### Acceptance criteria status
- [x] PanelLayout table defined with tree structure column (JSON column with `PanelNodeSchema`)
- [x] Events defined: LayoutSplit, LayoutPaneClosed, LayoutPaneAssigned, LayoutRestored
- [x] Materializer correctly updates layout tree from events (upsert via `onConflict("id", "replace")`)
- [ ] Tests: commit split event → verify tree structure updated (deferred — LiveStore store/adapter setup not yet available; will be testable after Issue #16)

### Issues unblocked
- Issue #73: PanelManager — serialize layout to LiveStore (partially — still needs #69)

### Design decisions
- Used `State.SQLite.json({ schema: PanelNodeSchema })` instead of plain `text` column with manual JSON.stringify/parse. This is the idiomatic LiveStore approach — the json column type automatically wraps with `Schema.parseJson()` for type-safe serialization.
- All layout events carry the full tree rather than incremental diffs. This keeps materializers simple (just upsert the whole row) and avoids complex tree-diffing in SQLite. The tradeoff is slightly larger event payloads, but panel layout trees are small (even deeply nested layouts are <10KB).
- Events are `synced` (not `clientOnly`) so layout state can be synced between server and client via LiveStore's sync mechanism, enabling session persistence across server restarts.

## Issue 29: PortAllocator service — allocate method — COMPLETED

Date: 2026-03-04

### What was done
- Created `packages/server/src/services/port-allocator.ts` with the `PortAllocator` Effect service using `Context.Tag("@laborer/PortAllocator")`.
- Implemented `allocate()` method using `Effect.fn` for call-site tracing:
  1. Uses an `Effect.Ref<Set<number>>` for fiber-safe concurrent access to allocated ports. This prevents race conditions when multiple workspaces are created simultaneously (sets up for Issue #32).
  2. Atomically finds the next unallocated port via `Ref.modify` — scans from `rangeStart` to `rangeEnd` and claims the first available port in a single atomic operation.
  3. Returns the allocated port number on success.
  4. Returns a descriptive `RpcError` with code `PORT_EXHAUSTED` when all ports in the range are allocated (sets up for Issue #31).
- Created `PortAllocator.make(rangeStart, rangeEnd)` factory for constructing a layer with a specific port range — useful for testing with smaller ranges.
- Created `PortAllocator.layer` as the default layer that reads `PORT_RANGE_START` and `PORT_RANGE_END` from `@laborer/env/server` using lazy import to avoid import-time side effects during testing.
- Used `Layer.unwrapEffect` to convert the effectful layer construction into a proper `Layer`.
- Verified: `tsc --noEmit` passes (no type errors — only pre-existing Effect plugin hints), `biome check` passes with zero issues on port-allocator.ts.

### Acceptance criteria status
- [x] PortAllocator is a tagged Effect service
- [x] `allocate()` returns a port within the configured range
- [x] Sequential allocations return unique ports (Ref.modify is atomic, ensuring uniqueness)
- [x] Allocated ports are tracked to prevent double-allocation (via Ref<Set<number>>)
- [ ] Tests: allocate returns port in range; two allocations return different ports (deferred — vitest not yet configured)

### Design decisions
- Used `Ref<Set<number>>` instead of a simple mutable `Set` for fiber-safety. `Ref.modify` provides atomic read-modify-write semantics, ensuring that even if multiple fibers call `allocate()` concurrently, they will never receive the same port. This satisfies Issue #32's requirements proactively.
- Used a linear scan from `rangeStart` to `rangeEnd` for simplicity. With a typical range of 900 ports (3100-3999), this is negligible overhead even at exhaustion.
- Separated `make` (accepts explicit range) from `layer` (reads from env) to enable testing with controlled port ranges without env dependency.
- The `-1` sentinel value pattern in `Ref.modify` avoids needing to throw inside the modify callback — errors are handled cleanly in the Effect pipeline via `Effect.flatMap`.

## Issue 43: WorkspaceProvider — destroy worktree — COMPLETED

Date: 2026-03-04

### What was done
- Added `destroyWorktree(workspaceId)` method to the `WorkspaceProvider` service interface and implementation in `packages/server/src/services/workspace-provider.ts`.
- Implementation uses `Effect.fn("WorkspaceProvider.destroyWorktree")` for call-site tracing:
  1. Looks up the workspace in LiveStore's `workspaces` table using `Array.findFirst` and `pipe`.
  2. If no workspace found, returns `RpcError` with code `NOT_FOUND`.
  3. Looks up the project via `ProjectRegistry.getProject` to get the repo path for git commands.
  4. Commits `workspaceStatusChanged` event to LiveStore with status "destroyed" immediately (so the UI reflects the state change even if cleanup takes time).
  5. Runs `git worktree remove --force <worktreePath>` to remove the worktree directory. Uses `--force` to handle cases where the worktree has uncommitted changes.
  6. Runs `git branch -D <branchName>` to delete the workspace branch.
  7. Frees the allocated port via `PortAllocator.free()`.
  8. Commits `workspaceDestroyed` event to LiveStore (removes the row from the workspaces table).
- Resilient error handling: git worktree remove and git branch delete failures are logged as warnings but do not abort the cleanup process. This handles cases where the worktree directory or branch was manually deleted. Port free failures are also caught and logged.
- Updated `packages/server/src/rpc/handlers.ts`: replaced the `workspace.destroy` stub with a real handler that delegates to `WorkspaceProvider.destroyWorktree(workspaceId)`.
- Verified: `tsc --noEmit` passes (no type errors — only pre-existing Effect plugin hints), `biome check` passes with zero issues on both modified files.

### Acceptance criteria status
- [x] `destroyWorktree(workspaceId)` removes the worktree directory (via `git worktree remove --force`)
- [x] Branch is deleted from git (via `git branch -D`)
- [x] `git worktree list` no longer includes the worktree (after successful `git worktree remove`)
- [ ] Tests: create then destroy -> directory gone, branch gone, not in worktree list (deferred — vitest not yet configured)

### Design decisions
- Used `--force` flag on `git worktree remove` to handle dirty worktrees (uncommitted changes). Without `--force`, git refuses to remove worktrees with modifications, which would leave stale workspaces that can't be cleaned up.
- Made cleanup resilient: each step (worktree remove, branch delete, port free) catches errors independently. If one fails, the others still execute. This prevents partial cleanup where e.g. the worktree is removed but the port is never freed.
- Status is updated to "destroyed" before physical cleanup begins. This gives the UI immediate feedback while the actual git operations run.
- Committed `workspaceDestroyed` event at the end to remove the row from the table after all cleanup is done.

## Issue 83: DiffService — poll on interval — COMPLETED

Date: 2026-03-04

### What was done
- Extended the `DiffService` Effect service in `packages/server/src/services/diff-service.ts` with 4 new methods for polling management:
  - `startPolling(workspaceId, intervalMs?)` — starts a background polling fiber that runs `getDiff` on a configurable interval (default 2000ms). Errors during individual polls are logged as warnings but do not stop the polling loop. Calling `startPolling` on an already-polling workspace is a no-op.
  - `stopPolling(workspaceId)` — interrupts the polling fiber for a workspace and removes it from the active polling map. If not polling, this is a no-op.
  - `stopAllPolling()` — interrupts all active polling fibers and clears the map. Used during graceful shutdown.
  - `isPolling(workspaceId)` — returns whether a workspace is currently being polled.
- Used `Effect.Ref<Map<string, Fiber.RuntimeFiber<void, never>>>` for fiber-safe concurrent tracking of active polling fibers per workspace.
- Polling is implemented as `getDiff(workspaceId) |> Effect.catchAll(logWarning) |> Effect.repeat(Schedule.spaced(interval)) |> Effect.asVoid |> Effect.forkDaemon`. The daemon fiber runs independently of the calling fiber's scope.
- `stopPolling` uses `Ref.modify` for atomic removal of the fiber from the map before interrupting it, preventing race conditions.
- Added imports: `Duration`, `Fiber`, `Ref`, `Schedule` from `effect`.
- Defined `DEFAULT_POLL_INTERVAL_MS = 2000` constant (PRD specifies 1-2 seconds; defaulting to 2s).
- Updated the service interface to include all 4 new methods in the `DiffService` Context.Tag type.
- Updated module doc comment to reflect Issue #83 and document the polling API with usage examples.
- Verified: `tsc --noEmit` passes for server package (only pre-existing Effect plugin informational messages), `ultracite check` passes with zero issues on diff-service.ts.

### Acceptance criteria status
- [x] Polling runs at configurable interval (default 2 seconds) — via `Schedule.spaced(Duration.millis(intervalMs))`, default 2000ms
- [x] DiffUpdated event committed to LiveStore when diff content changes — `getDiff` commits the event on each poll
- [ ] Tests: modify file → DiffUpdated event within polling interval; verify periodic execution (deferred — vitest not yet configured)

### Design decisions
- Used `Effect.forkDaemon` instead of `Effect.fork` to ensure polling fibers outlive the calling fiber's scope. The polling should continue until explicitly stopped via `stopPolling` or `stopAllPolling`.
- Errors during individual polls are caught with `Effect.catchAll` and logged as warnings. This prevents transient errors (e.g., workspace being destroyed mid-poll) from killing the entire polling loop.
- The `stopAllPolling` method exists for graceful shutdown scenarios (Issue #128/129/130). The server can call this on SIGINT/SIGTERM to clean up all polling fibers.
- No deduplication yet (that's Issue #84). Every poll commits a DiffUpdated event even if the content hasn't changed. This is intentional — deduplication is a separate concern.
- The `isPolling` method is useful for debugging and for future workspace lifecycle integration (Issue #85) to check if polling is already active before starting.

- Terminal output events (`v1.TerminalOutput`) are committed by the server's `TerminalManager` service when node-pty's `onData` fires.
- These events sync to the client via the LiveStore WebSocket sync mechanism (`makeWsSync` in the dedicated worker).
- The `store.events({ filter: ["v1.TerminalOutput"] })` API streams confirmed events from the event log. Events are confirmed after the sync round-trip (server → sync backend → client pull).
- The `v1.TerminalOutput` materializer is a no-op (`() => []`) — output data is ephemeral and consumed directly by event stream subscribers, not stored in a table row. This keeps the SQLite database compact.

### Design decisions
- Used `store.events({ filter: ["v1.TerminalOutput"] })` instead of a table query because terminal output events have a no-op materializer — the data is ephemeral and not stored in any table. The `events()` API streams from the event log, which is the only way to access these events on the client.
- Used `AbortController` to signal the event stream to stop on component unmount. The `for await...of` loop checks `abortController.signal.aborted` on each iteration.
- Used `ResizeObserver` instead of a window `resize` event listener. `ResizeObserver` is more precise — it fires when the specific container element changes size, which is essential for allotment pane resizing (not just window resizing).
- Set `scrollback: 10000` as a reasonable default buffer size. This can be tuned later (Issue #127 tests 100k+ lines).
- Used a dark theme that matches zinc-950/zinc-50 colors from the app's Tailwind/shadcn theme. Terminal colors (red, green, yellow, etc.) use Tailwind v4 palette values for visual consistency.
- The component is intentionally focused on output rendering only (Issue #60). Input handling (Issue #61) and resize propagation to the server PTY (Issue #62) will be added in separate issues.
- WebGL addon is loaded inside a try-catch because it requires a browser with WebGL2 support. Mobile browsers and some testing environments may not support it. The fallback canvas renderer provides identical functionality, just with less GPU acceleration.
- Font stack includes JetBrains Mono, Fira Code, Cascadia Code, and system monospace fonts — covering the most common developer fonts.
