- Implemented the subscribe-before-serialize pattern in `terminal-ws.ts`:
  1. Subscribe to live PTY output FIRST, queuing any data that arrives during serialization
  2. Serialize the headless terminal's screen state via `getScreenState()`
  3. Send `{"type":"screenState","data":"<VT sequences>"}` as the first data frame

## Issue 9: Frontend: Terminal Session Router (core) ‚Äî COMPLETED

Date: 2026-03-14

### What was done
- Created `apps/web/src/lib/terminal-session-router.ts` with `TerminalSessionRouter` class:
  - Centralized WebSocket stream manager that enforces exactly one WebSocket per terminal ID
  - `subscribe(terminalId, callbacks)` ‚Äî registers a subscriber with `onOutput`, `onScreenState`, `onStatus` callbacks. First subscriber opens the WebSocket. Returns an unsubscribe function.
  - `sendInput(terminalId, data)` ‚Äî sends user input to the terminal's WebSocket as a raw text frame
  - `getConnectionStatus(terminalId)` ‚Äî returns current WebSocket connection status (connecting/connected/disconnected)
  - `getTerminalStatus(terminalId)` ‚Äî returns terminal process status (running/stopped/restarted)
  - `hasSubscribers(terminalId)` ‚Äî checks if a session has active subscribers
  - `getSessionCount()` ‚Äî returns number of active sessions (for testing/debugging)
  - `dispose()` ‚Äî closes all WebSockets, clears all sessions, rejects future subscriptions
  - Cached screenState delivered to late subscribers via `setTimeout(0)` without server round-trip
  - Cached exit state delivered to late subscribers for terminals that already stopped
  - Screen state cache cleared on terminal restart
  - Last subscriber unsubscribing tears down the session (closes WebSocket, cancels timers)
  - Reconnection with exponential backoff (500ms initial, 2x factor, 30s max)
  - Max 3 consecutive failures before giving up (prevents infinite reconnection)
  - No reconnection when terminal is stopped
  - Flow control: character-count ack frames (`{"type":"ack","chars":N}`) every 5000 chars
  - JSON control message parsing: `status` (running/stopped/restarted) and `screenState` types
  - Non-JSON and unrecognized JSON messages treated as raw PTY output
  - Guard against stale WebSocket events (checks `session.ws !== ws`)
  - Guard against disposed router (rejects new subscriptions with no-op)
- Extracted and reused the message protocol from `use-terminal-websocket.ts`:
  - Same `parseControlMessage()` function for JSON control message detection
  - Same flow control ack protocol (CHAR_COUNT_ACK_SIZE = 5000)
  - Same exponential backoff parameters (500ms, 2x, 30s max, 3 failures)
- Created comprehensive test suite `apps/web/test/terminal-session-router.test.ts` with 32 tests:
  - Subscribe/WebSocket lifecycle (7 tests): returns unsubscribe, opens one WS, shares WS, separate WS per terminal, closes on last unsub, keeps open with remaining subs, re-creates after full unsub
  - Output broadcasting (2 tests): broadcasts to all subscribers, excludes unsubscribed

  - Status control messages (4 tests): broadcasts status, handles stopped with exit code, delivers cached exit state, clears screenState on restart
