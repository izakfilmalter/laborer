### Design decisions
- **Eye icon**: Used `Eye` from lucide-react ŌĆö semantically represents "review/inspect". Purple color (`text-purple-600 dark:text-purple-400`) distinguishes it from the other action buttons: green=start (Play), blue=create/write (FileText), purple=review (Eye), red=destroy (Trash). The color follows a semantic pattern that maps rlph workflow stages to distinct colors.
- **Dialog form**: Like "Write PRD", "Review PR" needs a parameter (PR number), so a dialog provides the focused input experience. This is consistent with all other parameterized actions in the codebase.
- **Input type="text" with inputMode="numeric"**: Used `type="text"` instead of `type="number"` to avoid the browser's native number input spinners (which are awkward for PR numbers). `inputMode="numeric"` still triggers the numeric keyboard on mobile. `pattern="[0-9]*"` provides an additional hint to the browser.
- **String-to-int parsing**: The form stores the PR number as a string (for input handling) and parses it to an integer on submit via `Number.parseInt`. This matches the RPC contract which expects `Schema.Int`.
- **Validation checks**: Three validations: (1) non-empty, (2) parses to a positive integer (`> 0`), (3) is a whole number (no decimals). This catches "abc", "-1", "0", "3.5", and empty string.
- **Button placement**: The ReviewPrForm button is placed after the "Start Ralph Loop" button. The visual order follows the rlph workflow: Write PRD ŌåÆ Start Loop ŌåÆ Review PR ŌåÆ Fix Findings. The review step comes after implementation.
- **onTerminalSpawned callback**: Same pattern as WritePrdForm ŌĆö auto-expands the collapsible terminal section so the user can see the new terminal in the sidebar.

**Date:** 2026-03-05

### Root cause (two bugs)

**Bug 1 ŌĆö Committed changes invisible:** The DiffService only ran `git diff` (unstaged) and `git diff --staged` (staged). **Bug 2 ŌĆö PatchDiff crashes on multi-file diffs:** `PatchDiff` from `@pierre/diffs/react` throws `"FileDiff: Provided patch must contain exactly 1 file diff"` when the raw git diff contains changes to multiple files. With committed changes now included, the diff output is always multi-file.

### Diagnosis

Added comprehensive diagnostic logging to `DiffService.getDiff` that revealed:
- `git diff`: stdoutLen=262 (small unstaged change)
- `git diff --staged`: stdoutLen=0 (nothing staged)

- `combinedDiffLen=262` ŌĆö the service was blind to 28KB of committed changes

### Fix

**DiffService (`packages/server/src/services/diff-service.ts`):**
- Changed diff strategy from `git diff` + `git diff --staged` to `git diff <merge-base>`.
- The service now computes the merge-base between the worktree branch and the default branch (tries main, master, develop, origin/main, origin/master in order).
- `git diff <merge-base>` compares the merge-base commit against the **working tree**, capturing committed + staged + unstaged changes in a single command.
- Falls back to the old unstaged+staged approach when no merge-base is found (e.g., if the worktree branch IS the default branch).
- Extracted `spawnGit()` helper to reduce code duplication across git commands.

- Replaced `PatchDiff` (single-file only) with `parsePatchFiles()` + `FileDiff` (per-file rendering).
- `parsePatchFiles(diffContent)` from `@pierre/diffs` parses the raw multi-file git diff into an array of `FileDiffMetadata` objects.
- Each file is rendered with `<FileDiff fileDiff={...} options={...} />` from `@pierre/diffs/react`.
- All existing UX features preserved: `useTransition` for deferred rendering, MutationObserver for scroll position preservation, "Updated" flash indicator, last updated timestamp footer.

## Issue #129: Graceful shutdown ŌĆö persist LiveStore state ŌĆö COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Added explicit shutdown finalization and startup restoration logging to the LaborerStore service. On startup, entity counts are logged to confirm state was successfully restored from SQLite persistence. On shutdown (SIGINT/SIGTERM), an explicit `Effect.addFinalizer` logs final entity counts, calls `store.shutdown()` to flush pending writes and close the LiveStore client session, and logs the result. This makes the LiveStore persistence lifecycle observable and ensures clean shutdown ordering.

### What was done

Modified `packages/server/src/services/laborer-store.ts`:

1. **Startup entity count logging**: After `createStore()` completes (which materializes state from the persisted SQLite eventlog), queries the 4 primary tables (projects, workspaces, terminals, tasks) and logs their row counts. This confirms state was successfully restored from the previous session. Example output: `[LaborerStore] Store initialized ŌĆö restored from SQLite: 2 project(s), 5 workspace(s), 3 terminal(s), 8 task(s)`.

2. **Explicit shutdown finalizer via `Effect.addFinalizer`**: When the layer scope is torn down (SIGINT/SIGTERM ŌåÆ `BunRuntime.runMain` interrupts ŌåÆ `Layer.launch` tears down scopes), the finalizer:
   - Logs "Shutdown: flushing LiveStore state to SQLite..."
   - Queries and logs final entity counts for post-mortem verification
   - Calls `store.shutdown()` to flush pending event commits, close the client session (stops sync fibers), and close the internal `lifetimeScope` (which cascades to adapter cleanup)
   - Wraps `store.shutdown()` in `catchAll` to ensure shutdown continues even if the store is already in a degraded state (e.g., already shut down by another path)
   - Logs "Shutdown: LiveStore state persisted to SQLite successfully"

3. **Updated imports**: Added `tables` import from `@laborer/shared/schema` for startup/shutdown entity count queries.

4. **Updated JSDoc**: Added comprehensive documentation covering the persistence & shutdown lifecycle, shutdown ordering, and references to Issue #129.

### Files modified
- `packages/server/src/services/laborer-store.ts` ŌĆö Added startup logging, shutdown finalizer, `tables` import, updated JSDoc

### Shutdown ordering verification
The layer composition in `main.ts` ensures correct ordering:
- **TerminalManager** (line 136) is provided BEFORE LaborerStoreLive (line 143), so its finalizer runs FIRST ŌåÆ commits final terminal status events to the store
- **LaborerStoreLive** (line 143) finalizer runs SECOND ŌåÆ flushes store, logs counts, calls store.shutdown()
- **ServerLive** (line 144) tears down LAST ŌåÆ stops the Bun HTTP server

This ordering is critical: TerminalManager needs the store to be alive when it commits "stopped" status events during its own shutdown. The store needs to be alive when it logs final counts and flushes. The HTTP server can shut down last since nothing depends on it during teardown.

### Design decisions
- **Explicit finalizer despite upstream handling**: The `@livestore/adapter-node` already registers `Effect.acquireRelease` finalizers that call `db.close()` on both SQLite databases. Our explicit finalizer adds three things: (1) observable logging so shutdown progress is visible in server output, (2) final entity count logging for post-mortem verification, (3) explicit `store.shutdown()` call which flushes pending writes before the adapter's own finalizers run. This belt-and-suspenders approach ensures data integrity.
- **catchAll around store.shutdown()**: The `store.shutdown()` call is wrapped in `Effect.catchAll` as a defensive measure. During abnormal shutdown (e.g., the store was already in an error state, or the scope is being interrupted), `shutdown()` might throw. The catchAll ensures the finalizer completes and subsequent cleanup (adapter finalizers, server teardown) proceeds.
- **Entity count logging**: Queries the 4 primary domain tables (projects, workspaces, terminals, tasks) ŌĆö not diffs or panelLayout, which are transient state that's less critical for persistence verification. The startup/shutdown count pair makes it easy to verify in logs that no data was lost during a restart cycle.
- **No explicit "flush" API**: LiveStore doesn't expose a separate "flush pending writes" API. `store.shutdown()` is the official mechanism for ensuring all pending commits are persisted. The upstream `createStore()` implementation closes the `lifetimeScope` which cascades through the adapter's `acquireRelease` to `db.close()`, and SQLite guarantees that `close()` flushes the WAL to the main database file.

### How directional navigation works

The layout tree encodes spatial relationships through split orientations:
- **Horizontal splits** lay children out left-to-right ŌåÆ navigated with ArrowLeft/ArrowRight
- **Vertical splits** lay children out top-to-bottom ŌåÆ navigated with ArrowUp/ArrowDown

Algorithm example with a layout like:
```
ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆŌö¼ŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöÉ
Ōöé     Ōöé     B     Ōöé
Ōöé  A  Ōö£ŌöĆŌöĆŌöĆŌöĆŌöĆŌö¼ŌöĆŌöĆŌöĆŌöĆŌöĆŌöż
Ōöé     Ōöé  C  Ōöé  D  Ōöé
ŌööŌöĆŌöĆŌöĆŌöĆŌöĆŌö┤ŌöĆŌöĆŌöĆŌöĆŌöĆŌö┤ŌöĆŌöĆŌöĆŌöĆŌöĆŌöś
```
Tree: `H-Split(A, V-Split(B, H-Split(C, D)))`

- From A, pressing Right ŌåÆ enters the right subtree from the left edge ŌåÆ B
- From B, pressing Down ŌåÆ C (first child of the horizontal split below)
- From D, pressing Left ŌåÆ C (adjacent in horizontal split)
- From C, pressing Up ŌåÆ B (adjacent in vertical split)
- From A, pressing Left ŌåÆ nothing (at the left edge, stops)

### Design decisions

1. **Added imports**: `existsSync` and `readFileSync` from `node:fs` for config file reading.

2. **Added `LaborerConfig` interface**: Defines the shape of `.laborer.json` ŌĆö currently supports `setupScripts` as an optional array of shell command strings.

5. **Added `runSetupScript(command, cwd, env)` function**: Executes a single shell command via `Bun.spawn(["sh", "-c", command])` in the worktree directory.    - Logs "Running setup script: <command>" at INFO level

   - On success: logs "Setup script completed successfully: <command>"
   - Returns the array of results up to and including the first failure

7. **Integrated into `createWorktree`**: After worktree verification (step 8) and before LiveStore commit:
   - Moved workspace ID generation earlier (step 9) so the ID is available for `LABORER_WORKSPACE_ID` env var
   - Reads project config via `readProjectConfig` (step 10)
   - If `setupScripts` is defined and non-empty, builds workspace env vars and calls `executeSetupScripts`
   - Checks results: if any script failed, sets workspace status to `"errored"` (instead of `"running"`)
   - Logs summary: "All N setup script(s) completed successfully" or "Workspace setup failed: script 'X' exited with code N"
   - LiveStore commit (step 11) uses the computed `workspaceStatus`

8. **Added module-level constants**: `CONFIG_FILE` (".laborer.json"), `logPrefix` ("WorkspaceProvider") for structured logging.

9. **Updated JSDoc**: Added setup script documentation, `.laborer.json` format example, and Issue #35 reference.

### Design decisions
- **`.laborer.json` over `rlphConfig`**: The PRD mentions setup scripts are "defined per project in rlphConfig", but `rlphConfig` is currently always `null` and has no mechanism for configuration entry. A `.laborer.json` file at the project root is more practical: it's version-controlled, editable by hand, and follows the convention of project-local config files (`.eslintrc.json`, `.prettierrc`, etc.). Future work can add UI for editing this config.
- **`sh -c` execution**: Scripts are executed via `sh -c <command>` rather than parsing into argv. This allows shell features (pipes, redirects, `&&` chains, env var expansion) which are essential for real setup scripts like `cp .env.example .env && bun install`.
- **Sequential execution with early exit**: Scripts run one at a time, stopping at the first failure. This is the expected behavior for setup scripts ŌĆö if `bun install` fails, there's no point running subsequent scripts that depend on installed packages.
- **Errored status, not rollback**: When a setup script fails, the workspace is created with status "errored" rather than rolled back. This design choice is intentional ŌĆö Issue #37 will implement full rollback (remove worktree, free port, clean branch). The current behavior lets the user see the workspace was created but setup failed, and they can investigate or retry.
- **`process.env` inheritance**: Scripts inherit the full `process.env` so they have access to PATH, HOME, and other system variables. Workspace-specific vars are merged on top, so they take precedence.
- **Config file resilience**: If `.laborer.json` is missing, unreadable, or contains invalid JSON, workspace creation proceeds without running scripts. This ensures the config file is optional and non-breaking ŌĆö projects that don't need setup scripts don't need to create the file.
- **Structured logging**: All log messages use `Effect.annotateLogs("module", logPrefix)` for consistent structured logging, matching the pattern used by TerminalManager, DiffService, and other services.

## Issue #37: WorkspaceProvider ŌĆö handle setup script failure ŌĆö COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Implemented full rollback when a setup script fails during workspace creation. Previously (Issue #35), a failed setup script would set the workspace status to "errored" but leave the worktree, branch, and allocated port intact ŌĆö a resource leak. Now, when any setup script exits with a non-zero code, the workspace is fully rolled back: worktree directory removed, git branch deleted, port freed. The error returned to the caller includes the failed script's stdout and stderr output for user visibility.

### What was done

Modified `packages/server/src/services/workspace-provider.ts`:

1. **Added `SetupResult` discriminated union type**: `{ _tag: "Success" }` or `{ _tag: "Failure", command, exitCode, stdout, stderr }`. Replaces the previous approach of checking `workspaceStatus` variable inline.

3. **Added `buildSetupFailureMessage(failure)` function**: Builds a descriptive error message from the failed script's command, exit code, stdout, and stderr. Output is only included if non-empty.

4. **Added `rollbackWorktree(repoPath, worktreePath, branchName, port, portAllocator)` function**: Best-effort cleanup that runs three steps sequentially:
   - `git worktree remove --force <worktreePath>` ŌĆö removes the worktree directory
   - `git branch -D <branchName>` ŌĆö deletes the branch
   - `portAllocator.free(port)` ŌĆö frees the allocated port

   - Calls `rollbackWorktree` for full cleanup
   - Commits `workspaceDestroyed` to remove the errored workspace from LiveStore
   - Returns `RpcError` with code `SETUP_SCRIPT_FAILED` and descriptive message including script output
   When scripts succeed, the workspace is committed with status "running" as before.

6. **Removed `let workspaceStatus` variable**: The workspace status is now always "running" on success (scripts passed or no scripts configured). On failure, the function returns early with an error after rollback ŌĆö no workspace is created.

7. **Updated JSDoc**: Added Issue #37 reference and updated the setup script documentation to describe the rollback behavior.

2. **Added `buildDirtySummary(lines)` function**: Takes an array of porcelain status lines, counts each category via `categorizeStatusLine`, and produces a human-readable summary like "3 modified, 1 untracked". Categories are output in a fixed order (modified ŌåÆ added ŌåÆ deleted ŌåÆ untracked ŌåÆ other) for consistent messaging.

3. **Added `checkDirtyState(repoPath)` function**: Runs `git status --porcelain` in the repo directory via `Bun.spawn`. Returns `{ isDirty: boolean, summary: string }`. On non-zero exit code, returns `GIT_CHECK_FAILED` error. Empty output ŌåÆ `isDirty: false`. Non-empty output ŌåÆ `isDirty: true` with a summary built by `buildDirtySummary`.

4. **Integrated into `createWorktree` as step 3b**: After branch existence check (step 3) and before port allocation (step 4), calls `checkDirtyState(project.repoPath)`. If dirty, logs a warning and returns `DIRTY_WORKING_TREE` error with message: "Uncommitted changes prevent worktree creation ({summary}). Commit or stash your changes before creating a workspace."

5. **Updated JSDoc**: Added Issue #38 reference to the module documentation.

### Design decisions
- **Pre-check before resource allocation**: The dirty state check is placed at step 3b ŌĆö after project validation and branch check but BEFORE port allocation (step 4). This means no resources are allocated that would need cleanup if the check fails. No port to free, no worktree to remove, no branch to delete. This is the cleanest possible error path.
- **Proactive check vs. reactive handling**: While `git worktree add` often succeeds with a dirty working tree (since the new branch starts from the same HEAD), dirty state can lead to unexpected behavior ŌĆö agents working on inconsistent file state, merge conflicts when the worktree is eventually cleaned up, etc. The proactive check gives users a clear signal to commit or stash first.
- **`DIRTY_WORKING_TREE` error code**: A new, distinct error code that downstream code (Issue #49) can use to display targeted error messages in the UI. The code is semantic and self-documenting.
- **Human-readable summary**: Instead of just saying "dirty state detected", the error includes a categorized summary (e.g., "3 modified, 1 untracked") so the user knows the scope of the issue. The `buildDirtySummary` function uses a fixed output order for consistency.
- **Three extracted functions**: `categorizeStatusLine`, `buildDirtySummary`, and `checkDirtyState` are extracted as standalone functions rather than inline logic in `createWorktree`. This keeps `createWorktree`'s complexity below Biome's limit (max 20) and makes each function independently testable.
- **Warning log before error**: A warning is logged with the module annotation before returning the RPC error. This ensures the dirty state detection is visible in server logs for debugging, even if the client doesn't display the full error message.
