## Issue #4: Lifecycle phase enum and context — COMPLETED

### What was done
- Created `apps/web/src/components/lifecycle-phase-context.tsx` with:
  - `LifecyclePhase` const enum: Starting (1), Ready (2), Restored (3), Eventually (4)
  - `LifecyclePhaseProvider` context provider with forward-only phase transitions
  - `useLifecyclePhase()` hook returning current phase, `advanceTo()`, and `when()`
  - Barrier pattern (inspired by VS Code) for `when(phase): Promise<void>` API
  - Phase transitions are irreversible — `advanceTo` with a lower phase is a no-op
  - Barriers for skipped intermediate phases are opened when jumping forward
- Wired `LifecyclePhaseProvider` into `apps/web/src/routes/__root.tsx` as the outermost provider (above ThemeProvider and all other providers)
- Created `apps/web/test/lifecycle-phase-context.test.tsx` with 9 tests:
  1. Initial phase is Starting
  2. advanceTo(Ready) changes phase to Ready
  3. advanceTo(Starting) after Ready is a no-op (no regression)
  4. advanceTo(Eventually) skips intermediate phases (jumps forward)
  5. Multiple advanceTo calls with the same phase are idempotent
  6. when(phase) resolves when phase is reached
  7. when(phase) resolves immediately if phase already passed
  8. when(phase) resolves for skipped intermediate phases
  9. Provides sensible defaults without a provider

## Issue #5: `useWhenPhase` hook and service status hook — COMPLETED

### What was done
- Created `apps/web/src/hooks/use-when-phase.ts` with:
  - `useWhenPhase(targetPhase): boolean` — returns `true` once the current lifecycle phase is at or past the target phase, `false` before
  - Thin hook that reads `useLifecyclePhase().phase` and compares with `>=`
  - Primary API for phase-aware UI — components use this to conditionally enable features
- Created `apps/web/src/hooks/use-service-status.ts` with:
  - `useServiceStatus(): ServiceStatuses` — returns a reactive map of per-service health states
  - Aggregates sidecar statuses from `useSidecarStatuses()` (server, terminal, file-watcher, mcp)
  - Adds a `sync` entry for LiveStore sync status (initially `unknown`, to be wired in Issue #2)
  - Exports `ServiceName` type (`'server' | 'terminal' | 'file-watcher' | 'mcp' | 'sync'`)
  - Exports `ServiceStatuses` type (`Record<ServiceName, ServiceState>`)
- Created `apps/web/test/lifecycle-phase-hooks.test.tsx` with 7 tests:
  - useWhenPhase:
    1. Returns false for Ready during Starting phase and true after advancing to Ready
    2. Returns true immediately for Starting (always past)
    3. Returns false for Eventually until Eventually phase is reached
  - useServiceStatus:
    4. Shows starting state for sidecar services initially
    5. Includes sync status entry with unknown state initially
    6. Updates sidecar status to healthy when health endpoint responds
    7. Updates reactively when service status changes

## Issue #1: Switch LiveStore to non-blocking sync — COMPLETED

### What was done
- Removed `initialSyncOptions: { _tag: 'Blocking', timeout: 5000 }` from `apps/web/src/livestore.worker.ts`
  - LiveStore now defaults to `{ _tag: 'Skip' }` — the store loads from the local OPFS cache immediately and syncs in the background
  - The `Suspense` boundary in `LiveStoreProvider` resolves in milliseconds (from OPFS) rather than waiting up to 5s for network sync
  - Background WebSocket sync starts automatically when the server is available
  - The `onSyncError` default (`'ignore'`) is kept, so sync failures don't crash the app
- Updated the file's JSDoc comment to document the non-blocking sync behavior
