### Architecture: Push events pattern
This is the first feature introducing push events (unsolicited) from Ghostty Host → GhosttyBridge → renderer, as opposed to the existing request/response pattern. The flow:
1. Ghostty runtime fires RuntimeAction callback during appTick()
2. Actions are queued into g_action_queue (thread-safe vector with mutex)
3. After appTick(), drainAndEmitActions() drains the queue and emits typed JSON events to stdout
4. GhosttyBridge in the main process detects push events (type in PUSH_ACTION_TYPES set)
5. Push events are forwarded to all renderer windows via BrowserWindow.getAllWindows().webContents.send()
6. Renderer receives via ipcRenderer.on() registered in onGhosttyAction preload method
7. React component subscribes via useEffect and filters by surfaceId

### Key decisions
- Polling/drain pattern chosen over ThreadSafeFunction: actions are queued during appTick() and drained synchronously afterwards, avoiding cross-thread N-API complexity
- surfaceId=0 is valid for early init actions: actions that fire during ghostty_surface_new() before the surface is registered get surfaceId=0 because we cannot pre-populate the reverse map without the pointer
- Push events use BrowserWindow.getAllWindows() broadcast: all renderer windows receive all action events, and the React component filters by surfaceId
- Bell notification uses a visual flash (ring style) rather than an audio beep — minimal and accessible
- child_exited and close_window both set a destroyed state on the pane, preventing further interaction
- cell_size events are received but not acted on (no-op) — the renderer controls sizing, not Ghostty
- renderer_health events log a warning — full health monitoring deferred to Issue 8

### Key decisions
- IOSurfaceRef is transmitted as a base64-encoded Buffer over JSON IPC: the raw pointer is process-local but Electron's sharedTexture API handles cross-process GPU texture sharing internally
