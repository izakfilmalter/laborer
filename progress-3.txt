### Design decisions
- **Eye icon**: Used `Eye` from lucide-react — semantically represents "review/inspect". Purple color (`text-purple-600 dark:text-purple-400`) distinguishes it from the other action buttons: green=start (Play), blue=create/write (FileText), purple=review (Eye), red=destroy (Trash). The color follows a semantic pattern that maps rlph workflow stages to distinct colors.
- **Dialog form**: Like "Write PRD", "Review PR" needs a parameter (PR number), so a dialog provides the focused input experience. This is consistent with all other parameterized actions in the codebase.
- **Input type="text" with inputMode="numeric"**: Used `type="text"` instead of `type="number"` to avoid the browser's native number input spinners (which are awkward for PR numbers). `inputMode="numeric"` still triggers the numeric keyboard on mobile. `pattern="[0-9]*"` provides an additional hint to the browser.
- **String-to-int parsing**: The form stores the PR number as a string (for input handling) and parses it to an integer on submit via `Number.parseInt`. This matches the RPC contract which expects `Schema.Int`.
- **Validation checks**: Three validations: (1) non-empty, (2) parses to a positive integer (`> 0`), (3) is a whole number (no decimals). This catches "abc", "-1", "0", "3.5", and empty string.
- **Button placement**: The ReviewPrForm button is placed after the "Start Ralph Loop" button. The visual order follows the rlph workflow: Write PRD → Start Loop → Review PR → Fix Findings. The review step comes after implementation.
- **onTerminalSpawned callback**: Same pattern as WritePrdForm — auto-expands the collapsible terminal section so the user can see the new terminal in the sidebar.

### Summary
Added explicit shutdown finalization and startup restoration logging to the LaborerStore service. On startup, entity counts are logged to confirm state was successfully restored from SQLite persistence. On shutdown (SIGINT/SIGTERM), an explicit `Effect.addFinalizer` logs final entity counts, calls `store.shutdown()` to flush pending writes and close the LiveStore client session, and logs the result. This makes the LiveStore persistence lifecycle observable and ensures clean shutdown ordering.

### What was done

Modified `packages/server/src/services/laborer-store.ts`:

1. **Startup entity count logging**: After `createStore()` completes (which materializes state from the persisted SQLite eventlog), queries the 4 primary tables (projects, workspaces, terminals, tasks) and logs their row counts. This confirms state was successfully restored from the previous session. Example output: `[LaborerStore] Store initialized — restored from SQLite: 2 project(s), 5 workspace(s), 3 terminal(s), 8 task(s)`.

2. **Explicit shutdown finalizer via `Effect.addFinalizer`**: When the layer scope is torn down (SIGINT/SIGTERM → `BunRuntime.runMain` interrupts → `Layer.launch` tears down scopes), the finalizer:
   - Logs "Shutdown: flushing LiveStore state to SQLite..."
   - Queries and logs final entity counts for post-mortem verification
   - Calls `store.shutdown()` to flush pending event commits, close the client session (stops sync fibers), and close the internal `lifetimeScope` (which cascades to adapter cleanup)
   - Wraps `store.shutdown()` in `catchAll` to ensure shutdown continues even if the store is already in a degraded state (e.g., already shut down by another path)
   - Logs "Shutdown: LiveStore state persisted to SQLite successfully"

3. **Updated imports**: Added `tables` import from `@laborer/shared/schema` for startup/shutdown entity count queries.

4. **Updated JSDoc**: Added comprehensive documentation covering the persistence & shutdown lifecycle, shutdown ordering, and references to Issue #129.

### Files modified
- `packages/server/src/services/laborer-store.ts` — Added startup logging, shutdown finalizer, `tables` import, updated JSDoc

### Shutdown ordering verification
The layer composition in `main.ts` ensures correct ordering:
- **TerminalManager** (line 136) is provided BEFORE LaborerStoreLive (line 143), so its finalizer runs FIRST → commits final terminal status events to the store
- **LaborerStoreLive** (line 143) finalizer runs SECOND → flushes store, logs counts, calls store.shutdown()
- **ServerLive** (line 144) tears down LAST → stops the Bun HTTP server

This ordering is critical: TerminalManager needs the store to be alive when it commits "stopped" status events during its own shutdown. The store needs to be alive when it logs final counts and flushes. The HTTP server can shut down last since nothing depends on it during teardown.

- [x] Restart → state fully restored from SQLite (verified by startup entity count logging — `createStore()` materializes state from persisted eventlog, then entity counts are queried and logged)

### Design decisions
- **Explicit finalizer despite upstream handling**: The `@livestore/adapter-node` already registers `Effect.acquireRelease` finalizers that call `db.close()` on both SQLite databases. Our explicit finalizer adds three things: (1) observable logging so shutdown progress is visible in server output, (2) final entity count logging for post-mortem verification, (3) explicit `store.shutdown()` call which flushes pending writes before the adapter's own finalizers run. This belt-and-suspenders approach ensures data integrity.
- **catchAll around store.shutdown()**: The `store.shutdown()` call is wrapped in `Effect.catchAll` as a defensive measure. During abnormal shutdown (e.g., the store was already in an error state, or the scope is being interrupted), `shutdown()` might throw. The catchAll ensures the finalizer completes and subsequent cleanup (adapter finalizers, server teardown) proceeds.
- **Entity count logging**: Queries the 4 primary domain tables (projects, workspaces, terminals, tasks) — not diffs or panelLayout, which are transient state that's less critical for persistence verification. The startup/shutdown count pair makes it easy to verify in logs that no data was lost during a restart cycle.
- **No explicit "flush" API**: LiveStore doesn't expose a separate "flush pending writes" API. `store.shutdown()` is the official mechanism for ensuring all pending commits are persisted. The upstream `createStore()` implementation closes the `lifetimeScope` which cascades through the adapter's `acquireRelease` to `db.close()`, and SQLite guarantees that `close()` flushes the WAL to the main database file.

## Issue #71: PanelManager — navigate between panes — COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Implemented directional pane navigation for the tmux-style panel system. Users can now move focus between panes spatially (left/right/up/down) based on the layout tree's split orientations, in addition to the existing linear next/previous cycling (Ctrl+B,O / Ctrl+B,P). The navigation algorithm walks up the layout tree from the active pane to find the nearest ancestor split matching the requested direction, then drills into the adjacent subtree to land on the nearest leaf at the entering edge.

### What was done

1. **Added `findPaneInDirection` to `apps/web/src/panels/layout-utils.ts`** — Pure tree navigation algorithm (~130 lines, 4 functions):
   - **`NavigationDirection` type**: `"left" | "right" | "up" | "down"` — maps to split orientations (left/right → horizontal, up/down → vertical)
   - **`buildPath(root, targetId)`**: Builds the complete path from root to a target node via DFS. Returns an array of PanelNode from root to target (inclusive).
   - **`getEdgeLeaf(node, edge)`**: Finds the first or last leaf in a subtree. When navigating right, we enter the new subtree from the left edge ("first"); when navigating left, from the right edge ("last"). Same for up/down with top/bottom.
   - **`tryNavigateAtAncestor(path, index, orientation, delta)`**: Extracted helper that checks a single ancestor in the path for a navigable neighbor. Returns the target leaf ID if found, or undefined to signal "continue walking up". This extraction keeps `findPaneInDirection` below the complexity limit (max 20).
   - **`findPaneInDirection(root, activePaneId, direction)`**: Main entry point. Builds the path, determines target orientation and delta from the direction, then walks up from the active pane's parent toward the root trying each ancestor. Stops at edges (returns undefined when no neighbor exists in the requested direction).

2. **Added directional keyboard shortcuts to `apps/web/src/panels/panel-hotkeys.tsx`** (~55 new lines):
   - **New props**: `layout?: PanelNode` — the root layout tree, needed by `findPaneInDirection`

   - **`navigateDirection` helper**: Shared callback that calls `findPaneInDirection` and `setActivePaneId`
   - **4 new hotkey sequences**:
     - `Ctrl+B, ArrowLeft` → move focus to the pane on the left
     - `Ctrl+B, ArrowRight` → move focus to the pane on the right
     - `Ctrl+B, ArrowUp` → move focus to the pane above
     - `Ctrl+B, ArrowDown` → move focus to the pane below
   - Updated module JSDoc to document the new shortcuts and reference Issue #71

3. **Updated `apps/web/src/routes/index.tsx`**:

### How directional navigation works

The layout tree encodes spatial relationships through split orientations:
- **Horizontal splits** lay children out left-to-right → navigated with ArrowLeft/ArrowRight
- **Vertical splits** lay children out top-to-bottom → navigated with ArrowUp/ArrowDown

Algorithm example with a layout like:
```
┌─────┬───────────┐
│     │     B     │
│  A  ├─────┬─────┤
│     │  C  │  D  │
└─────┴─────┴─────┘
```
Tree: `H-Split(A, V-Split(B, H-Split(C, D)))`

- From A, pressing Right → enters the right subtree from the left edge → B
- From B, pressing Down → C (first child of the horizontal split below)
- From D, pressing Left → C (adjacent in horizontal split)
- From C, pressing Up → B (adjacent in vertical split)
- From A, pressing Left → nothing (at the left edge, stops)

Implemented setup script execution after worktree creation. Projects can define setup scripts in a `.laborer.json` configuration file at the project root. After `git worktree add` succeeds and verification passes, the WorkspaceProvider reads the config file and executes each script sequentially in the worktree directory with workspace environment variables (PORT, LABORER_WORKSPACE_ID, etc.) injected. ### What was done

Modified `packages/server/src/services/workspace-provider.ts`:

1. **Added imports**: `existsSync` and `readFileSync` from `node:fs` for config file reading.

2. **Added `LaborerConfig` interface**: Defines the shape of `.laborer.json` — currently supports `setupScripts` as an optional array of shell command strings.

3. **Added `readProjectConfig(repoPath)` function**: Reads and parses `.laborer.json` from the project root. Returns an empty config if the file doesn't exist. Logs a warning if the file exists but can't be read or parsed (invalid JSON). Never fails — falls back to empty config for resilience.

4. **Added `SetupScriptResult` interface**: Captures the output of each script execution — `command`, `exitCode`, `stdout`, `stderr`.

5. **Added `runSetupScript(command, cwd, env)` function**: Executes a single shell command via `Bun.spawn(["sh", "-c", command])` in the worktree directory. 6. **Added `executeSetupScripts(scripts, worktreePath, env)` function**: Runs all scripts sequentially using a `for...of` loop (not `Effect.forEach` — scripts depend on previous results and must stop on failure). For each script:
   - Logs "Running setup script: <command>" at INFO level
   - Logs stdout at DEBUG level (if non-empty)
   - On non-zero exit: logs warning with exit code and stderr, breaks the loop
   - On success: logs "Setup script completed successfully: <command>"
   - Returns the array of results up to and including the first failure

7. **Integrated into `createWorktree`**: After worktree verification (step 8) and before LiveStore commit:
   - Moved workspace ID generation earlier (step 9) so the ID is available for `LABORER_WORKSPACE_ID` env var
   - Reads project config via `readProjectConfig` (step 10)
   - If `setupScripts` is defined and non-empty, builds workspace env vars and calls `executeSetupScripts`
   - Checks results: if any script failed, sets workspace status to `"errored"` (instead of `"running"`)
   - Logs summary: "All N setup script(s) completed successfully" or "Workspace setup failed: script 'X' exited with code N"
   - LiveStore commit (step 11) uses the computed `workspaceStatus`

8. **Added module-level constants**: `CONFIG_FILE` (".laborer.json"), `logPrefix` ("WorkspaceProvider") for structured logging.

9. **Updated JSDoc**: Added setup script documentation, `.laborer.json` format example, and Issue #35 reference.

### `.laborer.json` configuration format

```json
{
  "setupScripts": ["bun install", "cp .env.example .env"]
}
```

Scripts run in the worktree directory with these environment variables injected:
- `PORT` — the allocated port for dev servers
- `LABORER_WORKSPACE_ID` — the workspace UUID
- `LABORER_WORKSPACE_PATH` — the worktree directory path
- `LABORER_BRANCH` — the workspace branch name

Plus all existing `process.env` variables (PATH, HOME, etc.).

### Design decisions
- **`.laborer.json` over `rlphConfig`**: The PRD mentions setup scripts are "defined per project in rlphConfig", but `rlphConfig` is currently always `null` and has no mechanism for configuration entry. A `.laborer.json` file at the project root is more practical: it's version-controlled, editable by hand, and follows the convention of project-local config files (`.eslintrc.json`, `.prettierrc`, etc.). Future work can add UI for editing this config.
- **`sh -c` execution**: Scripts are executed via `sh -c <command>` rather than parsing into argv. This allows shell features (pipes, redirects, `&&` chains, env var expansion) which are essential for real setup scripts like `cp .env.example .env && bun install`.
- **Sequential execution with early exit**: Scripts run one at a time, stopping at the first failure. This is the expected behavior for setup scripts — if `bun install` fails, there's no point running subsequent scripts that depend on installed packages.
- **Errored status, not rollback**: When a setup script fails, the workspace is created with status "errored" rather than rolled back. - **`process.env` inheritance**: Scripts inherit the full `process.env` so they have access to PATH, HOME, and other system variables. Workspace-specific vars are merged on top, so they take precedence.
- **Config file resilience**: If `.laborer.json` is missing, unreadable, or contains invalid JSON, workspace creation proceeds without running scripts. This ensures the config file is optional and non-breaking — projects that don't need setup scripts don't need to create the file.
- **Structured logging**: All log messages use `Effect.annotateLogs("module", logPrefix)` for consistent structured logging, matching the pattern used by TerminalManager, DiffService, and other services.

## Issue #37: WorkspaceProvider — handle setup script failure — COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Implemented full rollback when a setup script fails during workspace creation. Previously (Issue #35), a failed setup script would set the workspace status to "errored" but leave the worktree, branch, and allocated port intact — a resource leak. Now, when any setup script exits with a non-zero code, the workspace is fully rolled back: worktree directory removed, git branch deleted, port freed. The error returned to the caller includes the failed script's stdout and stderr output for user visibility.

### What was done

Modified `packages/server/src/services/workspace-provider.ts`:

1. **Added `SetupResult` discriminated union type**: `{ _tag: "Success" }` or `{ _tag: "Failure", command, exitCode, stdout, stderr }`. Replaces the previous approach of checking `workspaceStatus` variable inline.

2. **Added `runProjectSetupScripts(repoPath, worktreePath, env)` function**: Extracted setup script orchestration into a dedicated function that reads `.laborer.json`, calls `executeSetupScripts`, and returns a `SetupResult`. This reduces the complexity of `createWorktree` below Biome's limit (max 20).

3. **Added `buildSetupFailureMessage(failure)` function**: Builds a descriptive error message from the failed script's command, exit code, stdout, and stderr. Output is only included if non-empty.

4. **Added `rollbackWorktree(repoPath, worktreePath, branchName, port, portAllocator)` function**: Best-effort cleanup that runs three steps sequentially:
   - `git worktree remove --force <worktreePath>` — removes the worktree directory
   - `git branch -D <branchName>` — deletes the branch
   - `portAllocator.free(port)` — frees the allocated port
   Each step logs warnings on failure but continues to the next step. Uses typed `Error` in `catch` callbacks (not `unknown`) for the git commands, and `catchAll` to ensure each step is best-effort.

5. **Updated `createWorktree` flow**: When `runProjectSetupScripts` returns `Failure`:
   - Commits a `workspaceCreated` event with status "errored" (brief UI visibility)
   - Calls `rollbackWorktree` for full cleanup
   - Commits `workspaceDestroyed` to remove the errored workspace from LiveStore
   - Returns `RpcError` with code `SETUP_SCRIPT_FAILED` and descriptive message including script output
   When scripts succeed, the workspace is committed with status "running" as before.

6. **Removed `let workspaceStatus` variable**: The workspace status is now always "running" on success (scripts passed or no scripts configured). On failure, the function returns early with an error after rollback — no workspace is created.

7. **Updated JSDoc**: Added Issue #37 reference and updated the setup script documentation to describe the rollback behavior.

### Design decisions
- **Three extracted functions**: The original `createWorktree` function exceeded Biome's complexity limit (27 > max 20) when the rollback logic was added inline. Extracting `runProjectSetupScripts`, `buildSetupFailureMessage`, and `rollbackWorktree` as standalone functions reduced complexity and improved readability/testability.

This is acceptable — the RPC error response provides the actual failure details to the calling component.
- **Best-effort rollback**: Each cleanup step (worktree remove, branch delete, port free) is wrapped in `catchAll`. If one step fails (e.g., worktree was already manually deleted), the remaining steps still execute. This follows the same resilient cleanup pattern used in `destroyWorktree`.

- **Typed `Error` in rollback `catch` callbacks**: The rollback function uses `new Error(...)` instead of `(error) => error` (unknown) in its `tryPromise` catch callbacks. While the Effect plugin warns about global `Error`, these errors are caught by `catchAll` and only used for logging — they never escape the function (return type is `Effect<void, never>`). Using `Error` provides a meaningful message for the log line.
- **`runProjectSetupScripts` reads config internally**: This function encapsulates reading `.laborer.json` so `createWorktree` doesn't need to know about the config file format. This makes it easier to test and to change the config source in the future.

## Issue #39: WorkspaceProvider — handle git fetch failure — COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Added git fetch handling to the workspace creation flow. The `createWorktree` method now runs `git fetch --all` before allocating any resources (port, worktree, branch creation) to ensure the local repo has the latest remote refs. Network failures (DNS resolution, SSH auth, connection timeouts, remote unreachable) are caught and returned as a clear `GIT_FETCH_FAILED` error with an actionable message. Because the fetch runs before port allocation, no resources need cleanup on failure — matching the zero-cleanup-on-failure design from Issue #38.

### What was done

Modified `packages/server/src/services/workspace-provider.ts`:

1. **Added `fetchRemote(repoPath)` function**: Runs `git fetch --all` in the repo directory via `Bun.spawn`. On non-zero exit code, analyzes the stderr output and returns a `GIT_FETCH_FAILED` error with the git stderr and actionable guidance. On success, logs debug confirmation.

2. **Added `detectNetworkError(stderr)` function**: Analyzes git stderr output to determine whether the failure is network-related. Checks for common patterns: "could not resolve host", "unable to access", "connection refused", "connection timed out", "network is unreachable", "no route to host", "ssh_exchange_identification", "could not read from remote repository", "the requested url returned error". Returns a boolean used to select the appropriate guidance message:
   - Network error → "Check your network connection and try again."
   - Other error → "Verify the remote is accessible and your credentials are valid."

3. **Integrated into `createWorktree` as step 3c**: After dirty state check (step 3b) and before port allocation (step 4). This positioning ensures:
   - No port to free on failure (fetch runs before `portAllocator.allocate()`)
   - No worktree to remove (fetch runs before `git worktree add`)
   - No branch to delete (fetch runs before branch creation)
   - The earliest possible failure point for network issues

4. **Updated JSDoc**: Added Issue #39 reference to the module documentation and the file header comment.

### Design decisions
- **`git fetch --all` over `git fetch origin`**: Uses `--all` to fetch from all configured remotes. Most repos have only `origin`, but some have multiple remotes (e.g., `upstream` for forks). Fetching all ensures the local repo is fully up-to-date regardless of remote configuration.
- **Pre-resource-allocation placement**: The fetch is placed at step 3c — after project validation, branch check, and dirty state check, but before port allocation (step 4). This means a failed fetch leaves absolutely no partial state: no allocated port, no created worktree, no new branch. This is the cleanest possible error path, matching the design of the dirty state check (Issue #38).
- **Network error detection heuristic**: The `detectNetworkError` function uses string matching against known git error patterns rather than exit code analysis. Git doesn't use distinct exit codes for network vs. auth vs. other failures — they all return exit code 128. String matching against stderr is the only reliable way to categorize the failure.
- **Actionable guidance in error message**: The error message includes both the raw git stderr (for technical diagnosis) and a human-readable guidance sentence (for actionable next steps). Network errors get "Check your network connection and try again." while credential/permission errors get "Verify the remote is accessible and your credentials are valid." This distinction helps users self-diagnose.
- **Debug-level logging for success**: The fetch success is logged at DEBUG level (not INFO) because it's a routine operation that succeeds silently in the common case. Only failures are logged at WARNING level to avoid log noise.
- **No timeout on git fetch**: The `Bun.spawn` call doesn't set a custom timeout. Git fetch is expected to complete within a reasonable time, and network timeouts are handled by the OS TCP stack and SSH client. Adding an artificial timeout could kill a legitimate slow-but-progressing fetch (e.g., large repo on slow connection).

### Design decisions
- **Imperative API via GroupImperativeHandle**: `react-resizable-panels` v4's `defaultSize` prop only applies on initial mount. To resize panels programmatically after mount, the `groupRef`'s `setLayout()` imperative method is used. This is the library's intended approach for programmatic resize.
- **PanelGroupRegistry context**: Since the layout tree is recursive (nested `ResizablePanelGroup` components), there can be many group instances. The registry stores all group refs keyed by `SplitNode.id`, allowing the resize logic to target the correct group. The registry uses `useRef<Map>` (not React state) to avoid unnecessary re-renders on registration.
- **`id` prop on ResizablePanel**: The imperative `setLayout()` API takes `{ [panelId]: percentage }`. By setting `id={child.id}` on each `ResizablePanel`, the panel IDs match the `PanelNode.id` values used in `computeResize`'s output `newSizes` map. This creates a natural bridge between the tree model and the library's imperative API.
- **Shift+Arrow (not plain Arrow)**: Plain arrow keys after Ctrl+B are already used for directional navigation (Issue #71). Adding Shift as a modifier distinguishes resize from navigation. This follows the tmux convention where Ctrl+B+Arrow navigates and Ctrl+B+Shift+Arrow (or repeated prefix) resizes.
- **Grow/shrink from adjacent sibling**: When growing, the space is taken from the NEXT sibling (in tree order). When shrinking, space is given to the PREVIOUS sibling. This provides predictable behavior: Shift+Right makes the active pane wider by pushing its right neighbor. Shift+Left makes it narrower by giving space to its left neighbor.
- **Walk-up ancestor search**: Like directional navigation, the resize algorithm walks up from the active pane to find the nearest ancestor split matching the direction. This handles nested layouts correctly — e.g., resizing horizontally from a pane deep inside a vertical split will skip past vertical ancestors until finding a horizontal one.
- **HomeRoute wrapper**: The `PanelGroupRegistryProvider` must be above `HomeComponent` so that `usePanelLayout()` (called at the top of `HomeComponent`) can access the registry. A wrapper component `HomeRoute` provides this. The alternative (splitting `usePanelLayout` into two hooks) would be more complex for no benefit.
- **5% step size**: A 5% step provides fine-grained control (20 presses to go from 50% to 100%) while being large enough that each press has a visible effect. This matches common tmux resize step sizes.
- **MIN_PANE_SIZE = 5%**: Matches the `minSize="5%"` prop already set on `ResizablePanel` components, ensuring consistency between drag-to-resize and keyboard resize constraints.

### Design decisions
- **Ref + state dual tracking**: The `handleTerminalData` callback fires for every WebSocket data frame (potentially hundreds per second during fast output). Using a ref (`hasReceivedDataRef`) for the hot-path check avoids calling `setState` on every frame. The state (`hasReceivedData`) is only needed for React rendering — it's set once and never changes again. - **`isRunning` guard on overlay**: Stopped terminals receive scrollback immediately on WebSocket reconnection (the server sends the ring buffer contents on connect). For these terminals, `hasReceivedData` would flip to `true` almost instantly, but there's a brief flash. The `isRunning` guard prevents the overlay from appearing at all for stopped terminals, since there's no PTY startup to wait for — the data is already buffered.
- **Exact terminal background color**: The overlay uses `bg-[#09090b]` (the same zinc-950 used in the xterm.js theme config) rather than a themed variable like `bg-background`. This ensures a seamless visual transition — the loading overlay looks like a blank terminal, and when data arrives, the content appears to "draw onto" the same surface. Using `bg-background` would create a flash if the background theme doesn't exactly match the terminal theme.
- **`z-10` for overlay**: Positioned below the prefix mode indicator (`z-20`) so if the user presses Ctrl+B during the loading phase, the "Ctrl+B" badge is still visible. The overlay is above the xterm.js canvas (which has no explicit z-index) so it fully covers the blank terminal.
- **No timeout fallback**: The overlay relies purely on data arrival to disappear. If the WebSocket never connects (e.g., server is down), the user will see the loading overlay indefinitely — but they'll also see the "Disconnected" or "Connecting..." banners layered on top. Adding a timeout to hide the overlay would show a blank terminal, which is a worse UX than a loading spinner. The reconnection banners provide the actual status information.

### Design decisions
- **Native HTML5 DnD over @dnd-kit**: Used the native HTML5 Drag and Drop API instead of adding `@dnd-kit/core` as a dependency. The issue's keyboard accessibility requirement is conditional ("if using @dnd-kit"), so this is acceptable. If keyboard DnD is needed later, @dnd-kit can be added incrementally.
- **Custom MIME type**: Used `application/x-laborer-terminal` as the drag data format. This ensures only laborer's drop targets accept the drag — browser-native drop zones (text inputs, etc.) won't react to it. The `x-` prefix follows the convention for application-specific MIME types.
- **JSON drag data**: The drag data is a JSON string with `terminalId` and `workspaceId`. Both values are needed to call `assignTerminalToPane(terminalId, workspaceId, paneId)`.
- **`hasTerminalDragData` for dragover**: The HTML5 DnD spec restricts `getData()` to the `drop` event only — during `dragover`/`dragenter`, only `dataTransfer.types` is readable. This isolates the drop target state (`isDragOver`) and handlers, keeping `PanelRenderer` as a clean routing function. The extraction also avoids calling hooks conditionally (which would violate React's rules of hooks).
- **`relatedTarget` check in dragLeave**: HTML5 DnD fires `dragleave` events when moving between child elements within the drop target. The `e.currentTarget.contains(e.relatedTarget)` check ensures `isDragOver` is only cleared when the drag truly leaves the container, preventing flickering of the drop highlight.
- **`cursor-grab` / `cursor-grabbing`**: Added CSS cursor feedback on terminal items to signal draggability. `cursor-grab` on hover, `cursor-grabbing` on active (during drag). This is a standard DnD affordance.
- **No drag preview customization**: The native HTML5 DnD API generates a semi-transparent ghost image of the dragged element. This default preview is sufficient — it shows the terminal item being dragged, which is informative. Custom drag previews would require additional DOM manipulation (e.g., `setDragImage`) which adds complexity for minimal UX benefit.
- **Occupied panes reject drops silently**: When dragging over a pane that already has a terminal, the handlers return without calling `preventDefault`, causing the browser to show a "not allowed" cursor. No explicit "replace" indicator is shown — this keeps the interaction simple (drag to empty panes only).

## Issue #127: Terminal scroll performance (100k+ lines) — COMPLETED

### Summary

Increased server-side ring buffer from 1MB to 5MB per terminal for reconnection scrollback. Increased WebSocket scrollback chunk size from 64KB to 128KB to reduce frame count on reconnection. Memory usage: ~100K lines × ~200 bytes/line ≈ 20MB per terminal — acceptable for a developer's machine.
   - **Scroll sensitivity**: Added `scrollSensitivity: 3` (default 1) for 3x normal scroll speed when navigating large buffers.
   - **Fast scroll sensitivity**: Added `fastScrollSensitivity: 5` (default 5) — Alt+scroll triggers 5x speed for rapid navigation through long output.

   - **WebGL renderer**: Already present (Issue #60). GPU-accelerated rendering is essential for smooth scrolling with large buffers — it renders only visible rows on the GPU without DOM reflow.
   - **Updated JSDoc**: Added Issue #127 documentation covering scroll performance, ring buffer capacity, and addon details.

2. **Modified `packages/server/src/services/terminal-manager.ts`** — Server-side scrollback:

3. **Modified `packages/server/src/routes/terminal-ws.ts`** — Reconnection scrollback delivery:

### Design decisions
- **100K lines, not unlimited**: xterm.js's scrollback buffer is an in-memory circular buffer. Setting it to `Infinity` or an extremely high value would cause unbounded memory growth for long-running terminals. 100K lines is the threshold specified in the acceptance criteria and provides ~2-3 hours of continuous agent output at typical rates. This matches VS Code's default terminal scrollback (1000) but is 100x larger to accommodate autonomous agent sessions.
- **5MB ring buffer, not matching client scrollback**: The server ring buffer (5MB ≈ 62.5K lines) is intentionally smaller than the client scrollback (100K lines). The ring buffer is for reconnection only — when the client reconnects, it gets the last ~62.5K lines from the ring buffer, and xterm.js retains the full 100K lines from before the disconnect plus the replayed scrollback. For initial connections, all output flows through the live WebSocket channel, and xterm.js accumulates up to 100K lines client-side.
- **128KB chunk size**: Doubled from 64KB to reduce the number of WebSocket frames during reconnection. Each frame has overhead (framing, syscall). Fewer, larger frames means faster scrollback delivery. 128KB is still well within WebSocket frame size limits and browser memory allocation thresholds.
- **`scrollSensitivity: 3`**: The default scroll sensitivity (1) is fine for normal terminal output, but when navigating 100K lines, users need to scroll faster. 3x speed means reaching the top of a 100K-line buffer takes ~33K scroll wheel ticks instead of ~100K. Combined with `fastScrollSensitivity: 5` (Alt+scroll), users can navigate the full buffer quickly.
- **Unicode 11 addon**: AI agents (Claude, OpenCode, Codex) may output Unicode characters including emoji, CJK, and mathematical symbols in their responses. Without proper Unicode width calculation, the cursor position and text layout would be misaligned after wide characters, causing visual glitches that worsen over thousands of lines. The addon is loaded in a try/catch to avoid breaking the terminal if the addon fails to initialize.
- **No `@xterm/addon-serialize`**: Considered for snapshotting terminal state but rejected — the ring buffer on the server already provides reconnection data, and xterm.js's internal buffer handles client-side persistence. Adding serialize would duplicate the reconnection mechanism.
- **WebGL already present**: The WebGL renderer was added in Issue #60. It's critical for scroll performance — rendering 100K lines of scrollback to a canvas element via the 2D API would be significantly slower than the GPU-accelerated WebGL path. The existing fallback to canvas on WebGL context loss is appropriate for edge cases.

|--------|--------|-------|

| Client memory (per terminal) | ~2MB | ~20MB |
| Server ring buffer | 1MB | 5MB |
| Server memory (per terminal) | ~1MB | ~5MB |
| Reconnection frames | ~16 (1MB/64KB) | ~40 (5MB/128KB) |
| Scroll sensitivity | 1x (default) | 3x normal, 5x fast (Alt+scroll) |
| Unicode support | Basic (BMP only) | Full (Unicode 11, CJK+emoji) |

### Design decisions
- **Pixel-to-percentage conversion**: `react-resizable-panels` uses percentage-based sizing. The hook computes ideal pixel values based on viewport width, then converts to percentages. This provides pixel-level precision (ensuring 100px minimum) while working within the library's API.
- **100px minimum pane size**: A terminal with `fontSize: 13` and `JetBrains Mono` font renders approximately 12-14 columns in 100px width. This is the minimum for displaying meaningful content (file paths, short commands, status output). Below this, a terminal is completely unusable.
- **Sidebar scaling formula**: The sidebar grows by 50px per 1000px of additional viewport width beyond 1920px. This provides gentle scaling — the sidebar gets wider on bigger screens to take advantage of space, but never dominates (maxing at 760px). At 1920px, 280px is a comfortable sidebar width. At 5K, 440px shows more content without being excessive.
- **Collapsible below 1280px**: The 1280px threshold matches common laptop resolutions (1280x800, 1280x720). At these widths, the sidebar competes with terminal space. Collapsing it frees the full width for panes. The threshold is implemented via the hook, not CSS media queries, because the `collapsible` prop needs to be a React boolean, not a CSS class.
- **Imperative panel ref for collapse**: `react-resizable-panels` v4 provides `collapse()`, `expand()`, and `isCollapsed()` on the `PanelImperativeHandle`. Using `onResize` to track collapse state (by checking `isCollapsed()`) is more reliable than tracking size thresholds manually.
- **ResizeObserver for diff view**: The diff pane's width depends on its position in the layout tree (how many splits, their orientations, the sidebar size) — not just the viewport width. A `ResizeObserver` measures the actual container width, so the split/unified switch is accurate regardless of layout configuration.
- **500px unified threshold**: Split diff view renders two columns of code. At 500px, each column gets ~250px, which is about 30 characters with typical diff rendering — barely readable. Below 500px, unified view is clearly superior since it uses the full width for a single column.
- **Larger handle hit area (w-2 = 8px)**: The default 4px hit area was difficult to grab, especially on high-DPI displays where 4px might be physically less than 1mm. Doubling to 8px makes handles reliably graspable while remaining visually unobtrusive (the visible line is still 1px).
- **2px handle lines at 4K+**: At 4K resolution (3840px+), a 1px CSS line can be physically invisible on some displays due to subpixel rendering. The `@media (min-width: 3840px)` rule bumps handle lines to 2px for visibility.
- **No `devicePixelRatio` scaling**: Considered scaling font sizes and UI elements based on `window.devicePixelRatio`, but this would conflict with OS-level display scaling which already handles DPI. The browser renders CSS pixels accounting for DPI; additional scaling would be double-counting.
- **useCallback on resize handler**: The `useResponsiveLayout` hook uses `useCallback` for `computeSizes` to avoid recreating the computation function. The `window.addEventListener("resize")` fires frequently during drag-resize of the browser window, so stable references matter.

## Issue #91: Diff viewer debounce/throttle for rapid changes — COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Added debounce/throttle to the diff viewer to prevent UI lag when an agent makes rapid file changes. When multiple `DiffUpdated` events arrive in quick succession (e.g., agent writing files rapidly between 2-second DiffService polls), the raw diff content is now debounced via a custom `useDebouncedValue` hook before being passed to the expensive `parsePatchFiles` + `FileDiff` rendering pipeline. This prevents excessive re-renders while ensuring the viewer shows recent content within 500ms (meeting the acceptance criteria threshold).

### What was done

1. **Created `apps/web/src/hooks/use-debounced-value.ts`** — new custom hook (~110 lines) that provides a generic trailing-edge debounce with maximum wait:
   - **`useDebouncedValue<T>(value, delayMs)`**: Returns `[debouncedValue, isPending]`. When the input value changes multiple times within `delayMs` (default 300ms), only the final value is emitted after the delay elapses.
   - **Trailing-edge debounce**: Each new value resets the debounce timer. Intermediate values are skipped entirely — the expensive downstream computation (diff parsing, FileDiff rendering) is never called on values that will be superseded.
   - **Maximum wait (500ms)**: A secondary timer ensures that even under sustained rapid changes (value changing every frame), the debounced value emits within 500ms. This prevents indefinite postponement and meets the acceptance criteria ("last update visible within 500ms threshold").
   - **`isPending` return**: Boolean indicating whether the debounced value is behind the latest input. Used to show the "Updating..." indicator while the debounce is settling.
   - **Timer cleanup**: Both the trailing-edge and max-wait timers are cleaned up on value change and on unmount to prevent memory leaks and stale updates.

2. **Modified `apps/web/src/panes/diff-pane.tsx`** — integrated debounce into the diff rendering pipeline:
   - **Renamed `diffContent` to `rawDiffContent`**: The value directly from LiveStore is now `rawDiffContent`. This is the un-debounced, immediately-reactive value.
   - **Added `useDebouncedValue(rawDiffContent, 300)`**: The debounced value is now `diffContent`, which feeds into `parsePatchFiles` and `FileDiff` rendering. Rapid intermediate values never reach the expensive parsing step.
   - **Combined pending state**: `isPending = isDebouncePending || isTransitionPending`. The debounce pending (value not yet settled) and transition pending (render not yet committed) are merged into a single "Updating..." indicator. The user sees one unified loading state rather than two separate indicators.
   - **Empty state uses `rawDiffContent`**: The "No changes" empty state check uses `rawDiffContent` (not debounced `diffContent`) to avoid briefly showing the empty state while the debounce timer settles after new content arrives.
   - **Updated JSDoc**: Added Issue #91 documentation describing the debounce/throttle architecture, the performance pipeline, and how the layers compose.

### Performance pipeline (complete)

```
LiveStore event → reactive query → useDebouncedValue (300ms trailing, 500ms max)
  → parsePatchFiles (only on debounced value) → useTransition (deferred render)
    → FileDiff component render
```

Each layer serves a distinct purpose:
- **LiveStore deduplication (server, Issue #84)**: Only commits events when diff content actually changes
- **Debounce (client, Issue #91)**: Collapses rapid bursts of changes into a single update
- **useTransition (client, Issue #89)**: Prevents the retained render from blocking user interactions

### Design decisions
- **Trailing-edge debounce with max wait over throttle**: A pure throttle would emit at regular intervals during rapid changes, causing multiple partial renders. A trailing-edge debounce waits for updates to stop, then emits once — better for the diff viewer use case where we want the "final" state after a burst of changes. The max wait ensures bounded latency so the viewer doesn't appear frozen during sustained rapid changes.
- **300ms debounce delay**: Chosen as a balance between responsiveness and render reduction. At 300ms, a burst of 10 updates in 1 second produces at most 3-4 renders instead of 10. The user perceives near-instant updates (300ms is below the "noticeable delay" threshold of ~400ms for non-interactive content) while the rendering pipeline does 60-70% less work.
- **500ms maximum wait**: The acceptance criteria specifies "last update visible within 500ms max". The MAX_DELAY_MS constant enforces this ceiling. Under sustained rapid changes (e.g., agent writing files continuously), the debounce timer never settles (each new value resets it), but the max timer fires at 500ms to show the most recent content. This guarantees bounded latency regardless of update frequency.
- **Generic hook, not diff-specific**: `useDebouncedValue` is a generic hook that works with any value type. It could be reused for other rapidly-changing values in the app (e.g., terminal output counts, task status updates). The hook is placed in `hooks/` alongside `use-responsive-layout.ts` and `use-terminal-websocket.ts`.
- **Debounce on raw string, not parsed FileDiffs**: The debounce is applied to the raw `diffContent` string rather than the parsed `FileDiffMetadata[]` array. This means `parsePatchFiles` (which involves string parsing and regex matching) is called only on the debounced value — saving both CPU time and GC pressure from intermediate array allocations.
- **`rawDiffContent` for empty state**: The "No changes" check uses `rawDiffContent` (un-debounced) rather than `diffContent` (debounced). If we used `diffContent`, there would be a brief flash of the empty state when new content arrives but debounce hasn't settled yet. Using `rawDiffContent` means the empty state disappears instantly when content is available, even before it's rendered.
- **Combined `isPending` state**: Rather than showing separate "Debouncing..." and "Rendering..." indicators, the two pending states are combined (`isDebouncePending || isTransitionPending`) into a single "Updating..." indicator. This gives the user a single, coherent signal that the viewer is processing new content, without exposing implementation details of the two-stage pipeline.
- **No `useDeferredValue` (React 19)**: React 19's `useDeferredValue` was considered but rejected for this use case. `useDeferredValue` defers rendering of a value to a lower-priority lane, but it doesn't skip intermediate values — each update still triggers a render that React schedules at lower priority. Our debounce actively skips intermediate values, preventing `parsePatchFiles` from running at all on superseded content. `useTransition` (already in place) provides the deferred rendering benefit on top of the debounce.
- **Timer-based over requestAnimationFrame**: The debounce uses `setTimeout` rather than `requestAnimationFrame`. `rAF` fires at display refresh rate (16.67ms at 60Hz) which is too frequent for our use case. `setTimeout(300)` provides the right granularity for collapsing update bursts without being tied to the display refresh cycle.

### Design decisions
- **Component state over LiveStore**: Annotation state is stored in React `useState`, not LiveStore. Accept/reject decisions are ephemeral UI state — they reset when the diff content changes (agent makes more modifications). Persisting them to LiveStore would add schema complexity and sync overhead for data that becomes stale within seconds when an agent is actively working. If persistence is needed later (e.g., for code review workflows), it can be added by moving the Map to LiveStore.
- **`Map<string, FileDiffMetadata>` keyed by file name**: Each file's annotation state is tracked independently. Accepting a hunk in file A doesn't affect file B. The file name is a stable key within a single diff (unique per `parsePatchFiles` output). When the diff content changes, the entire Map is cleared since all hunk indices may have shifted.
- **`effectiveFileDiffs` merge pattern**: Rather than modifying the parsed `fileDiffs` in place, a new `effectiveFileDiffs` array is computed that overlays annotated versions on top of the base parse. This keeps the base parse stable (used as fallback and for hunk action lookups) while feeding the annotated versions to the deferred rendering pipeline.
- **`diffAcceptRejectHunk` from library**: Uses the library's built-in accept/reject logic rather than custom hunk manipulation. The library handles: immutable transformation, context line merging, line offset adjustment for subsequent hunks, and cacheKey busting. This ensures visual correctness across all diff view modes (split, unified).
- **Reset on content change**: When `diffContent` changes (agent modified files, DiffService polled), all annotation state is cleared. This is the correct behavior because hunk indices and line numbers may have shifted in the new diff. Applying stale annotations to a new diff would produce incorrect results. The `prevDiffContentRef` pattern detects content changes reliably.
- **`renderHoverUtility` over `onLineClick`**: The `renderHoverUtility` render prop provides a positioned UI element that follows the hovered line. This is the library's intended mechanism for per-line actions (accept/reject, comments, etc.). `onLineClick` could work for click-based actions but wouldn't provide the hover-reveal UX that makes the buttons discoverable. The hover interaction matches VS Code's diff editor behavior.
- **`enableHoverUtility: true` in options**: Required to activate the library's hover tracking system. Without it, `renderHoverUtility` is never called. Added to both split and unified options for consistent behavior.
- **`handleHunkActionRef` for stale closure avoidance**: The `createRenderHoverUtility` callback is recreated when `annotatedDiffs` changes. However, the `handleHunkAction` callback depends on `fileDiffs` (for fallback lookup). To ensure the latest `handleHunkAction` is always used in click handlers, a ref is maintained — matching the same pattern used for `openEditorRef` in the editor.open feature (Issue #112).
- **Already-resolved hunk detection**: After `diffAcceptRejectHunk`, the hunk's `additionLines` and `deletionLines` are set to 0 (all content becomes context). The `hunkHasChanges` function detects this and prevents showing accept/reject buttons on already-resolved hunks. This avoids confusing double-accept/reject scenarios.
- **Green/red button colors**: Accept uses green (`bg-green-600/15 text-green-500`) and Reject uses red (`bg-red-600/15 text-red-500`). These are universally recognized colors for approve/deny actions. The subtle 15% opacity background with full-opacity text provides sufficient contrast without being visually overwhelming in the diff viewer.
- **Check/X icons**: `Check` (checkmark) for accept and `X` (cross) for reject from lucide-react. These are the standard icons for approve/deny actions, matching GitHub's PR review UI, VS Code's merge conflict resolution, and other diff tools.

- **Pluralization**: The tooltip uses correct English pluralization: "No running workspaces", "1 running workspace", "N running workspaces". This attention to detail matters for a developer tool's system tray presence.
