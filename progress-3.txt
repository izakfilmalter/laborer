## Issue #64: Terminal session reconnection ‚Äî DONE

### Summary
Implemented terminal session reconnection so that on page reload, running terminals are detected from LiveStore, xterm.js reconnects to the terminal's output stream (via event log replay), new output appears after reconnection, and stopped terminals are shown in read-only mode with a status banner.

### Key discovery
LiveStore's `store.events()` replays ALL historical events from the persisted eventlog (OPFS/SQLite) starting from ROOT. Since `v1.TerminalOutput` events have a no-op materializer (they don't update any table), they are still persisted in the event log. On page reload, the `for await` loop over `store.events({ filter: ["v1.TerminalOutput"] })` replays every historical terminal output event, effectively restoring the entire scrollback buffer automatically. This means no separate server-side buffer is needed for reconnection ‚Äî LiveStore handles it.

### Changes made
- Modified `apps/web/src/panes/terminal-pane.tsx`:
  - Added reactive terminal status tracking from LiveStore's `terminals` table via `store.useQuery(allTerminals$)` with a `useMemo` filter by `terminalId`. Status is either "running" or "stopped" (defaults to "stopped" if terminal row not found).
  - Added `isRunningRef` to gate keyboard input ‚Äî when the terminal is stopped, the `onData` callback returns early so keystrokes are silently dropped instead of being sent to a dead PTY.
  - Added a "Process exited ‚Äî terminal output preserved (read-only)" banner at the bottom of the terminal when status is "stopped". Uses absolute positioning with a semi-transparent `bg-muted/90 backdrop-blur-sm` overlay.
  - Added initial resize on mount (`fitAddon.fit()` + `terminal.resize` RPC) to re-sync PTY dimensions on reconnection, since the terminal pane may have different dimensions than the last session.
  - Added JSDoc documentation explaining the reconnection architecture: event log replay, status tracking, panel layout persistence, and the full data flow from server PTY to xterm.js.
  - Added inline comments on `store.events()` replay behavior and input gating logic.
- Verified: `tsc --noEmit` and `biome check` show only pre-existing errors in UI component files (calendar.tsx, sonner.tsx, etc.) ‚Äî no new errors introduced by this change.

### Acceptance criteria status
- [x] On page reload, running terminals are detected from LiveStore (`store.useQuery(allTerminals$)` reads terminal status from `terminals` table)
- [x] xterm.js reconnects to the terminal's output stream (`store.events()` replays historical `v1.TerminalOutput` events into xterm.js)
- [x] New output appears after reconnection (live events stream in after historical replay completes)
- [ ] Tests: spawn terminal ‚Üí reload page ‚Üí terminal still running, new output visible (deferred ‚Äî requires live server + web app integration test)

### Design decisions
- Used `store.useQuery(allTerminals$)` with client-side `useMemo` filter instead of a parameterized query, consistent with the pattern used in `TerminalList`, `WorkspaceList`, and `ProjectList` components throughout the codebase.
- Used `isRunningRef` (ref) instead of direct state check in the `onData` callback because the xterm.js `onData` handler is registered once during the effect setup and never re-registered ‚Äî it needs to read the latest value via a ref to avoid closure staleness.
- Defaulted terminal status to "stopped" when the terminal row is not found (`row?.status ?? "stopped"`), which is the safest default: a missing terminal should not accept input.
- The status banner uses `absolute inset-x-0 bottom-0` positioning to overlay the terminal without disrupting the xterm.js container layout.

## Issue 128: Graceful shutdown ‚Äî kill all terminals ‚Äî COMPLETED

Date: 2026-03-04

### What was done
- Added a graceful shutdown finalizer to the `TerminalManager` Effect service in `packages/server/src/services/terminal-manager.ts`.
- Changed `Layer.effect` to `Layer.scoped` for the `TerminalManager.layer` to support `Effect.addFinalizer`. Scoped layers have their finalizers invoked when the layer scope is torn down ‚Äî which happens automatically when `BunRuntime.runMain` receives SIGINT/SIGTERM and interrupts the main fiber.
- The finalizer runs when the server shuts down (SIGINT/SIGTERM) and performs:
  1. Takes a snapshot of all in-memory terminals via `Ref.get(terminalsRef)`.
  2. If no active terminals, logs "no active terminals to clean up" and returns early.
  3. Iterates all terminals using `Effect.forEach` with `{ discard: true }`:
     - Sends a kill command to the PTY Host via `ptyHostClient.kill(terminal.id)`.
     - Commits `terminalStatusChanged` event to LiveStore with status "stopped" to update the persistent state.
     - Increments the killed count.
  4. Each individual terminal kill is wrapped in `Effect.catchAll` ‚Äî errors are logged as warnings but do not abort the cleanup. This ensures best-effort resource recovery.
  5. Clears the in-memory terminals map via `Ref.set`.
  6. Logs the final count: "Shutdown: killed X/Y terminal(s)".
- The shutdown finalizer complements the existing stale terminal cleanup on startup (lines 181-198): on startup, any terminals with status "running" in LiveStore are marked as "stopped" since they're orphans from a previous session. Together, these two mechanisms provide belt-and-suspenders reliability:
  - Normal shutdown: finalizer kills all PTYs and updates LiveStore (clean state on restart).
  - Abnormal shutdown (crash/SIGKILL): stale terminal cleanup on next startup marks orphans as "stopped".
- The `PtyHostClient.layer` already has its own finalizer (line 298-306 of pty-host-client.ts) that kills the PTY Host child process on teardown. Since Effect tears down layers in reverse order of construction, the TerminalManager finalizer runs first (killing individual terminals with LiveStore updates), then the PtyHostClient finalizer runs (killing the PTY Host process which terminates any remaining PTY children). This ordering ensures both individual terminal cleanup AND PTY Host process cleanup happen reliably.
- Updated module JSDoc to reference Issue #128.
- Verified: `tsc --noEmit` passes (no type errors ‚Äî only pre-existing Effect plugin informational messages), `ultracite check` passes with zero issues on terminal-manager.ts.

### Acceptance criteria status
- [x] Shutdown signal ‚Üí all PTY processes terminated (finalizer iterates all in-memory terminals and kills each via PtyHostClient, then PtyHostClient's own finalizer kills the PTY Host child process)
- [x] No orphan processes after shutdown (PtyHostClient.kill sends kill command to PTY Host which terminates the PTY; PtyHostClient finalizer kills the PTY Host process itself; on next startup, stale terminal cleanup handles any edge cases)
- [ ] Tests: spawn terminals ‚Üí shutdown ‚Üí all processes gone (deferred ‚Äî vitest not yet configured; manual verification: SIGINT triggers finalizer logs)

### Design decisions
- Changed from `Layer.effect` to `Layer.scoped` to support `Effect.addFinalizer`. This is the standard Effect pattern for resources that need cleanup ‚Äî scoped layers have their scope automatically managed by the containing layer composition.
- Used the same best-effort cleanup pattern as `killAllForWorkspace` (Issue #44): each terminal kill catches errors independently. If one PTY is already dead or in a bad state, the others are still cleaned up.
- The finalizer logs at the "module" annotation level (`Effect.annotateLogs("module", logPrefix)`) for consistent structured logging that matches the rest of the TerminalManager's log output.
- The finalizer runs BEFORE the PtyHostClient finalizer (reverse layer construction order) which is correct: we want to send individual kill commands (which update LiveStore) before the PTY Host process is forcefully terminated. If the PtyHostClient finalizer ran first, the kill commands would fail because the PTY Host would already be dead.

- Used `useTransition` rather than `useDeferredValue` because we need both the `isPending` flag (for the loading indicator) and explicit control over when the transition starts. `useDeferredValue` only defers the value, with no way to show a pending state.
- The "Updated" flash uses `prevContentRef` (not `useRef` on the deferred content) to detect changes. This ensures the flash fires immediately when new content arrives from LiveStore, not when the transition completes. The user sees "Updated" while the PatchDiff is still re-rendering.
- Moved PatchDiff options to module level (`PATCH_DIFF_OPTIONS`). This is safe because all DiffPane instances use the same visual configuration. If per-instance options were needed in the future (e.g., unified vs. split mode toggle), the options could be wrapped in `useMemo`.
- The timestamp footer intentionally uses a simple relative format ("just now" / time string) rather than continuously updating relative times (e.g., "5 seconds ago"). Continuously updating would require a timer that adds complexity for minimal UX benefit ‚Äî the timestamp is informational, not interactive.

## Issue #138: Ring buffer data structure + unit tests ‚Äî COMPLETED

Date: 2026-03-04

### What was done
- Created `packages/server/src/lib/ring-buffer.ts` ‚Äî a standalone ring buffer (circular buffer) data structure for terminal scrollback, as specified in PRD-terminal-perf.md "Server-Side Ring Buffer for Scrollback" section.
- The ring buffer uses a `Uint8Array` with a configurable capacity (default 1MB) and a write cursor that wraps around when full.
- Created `packages/server/test/ring-buffer.test.ts` ‚Äî 32 comprehensive unit tests covering all acceptance criteria from PRD-terminal-perf.md "Ring Buffer Unit Tests" section.

### Implementation details
- **`RingBuffer` class** with the following API:
  - `constructor(capacity?: number)` ‚Äî default 1MB, validates positive capacity
  - `write(data: Uint8Array)` ‚Äî appends bytes, wrapping when full. Handles: data smaller than capacity (no wrap), data exactly at capacity boundary (wrap), data larger than capacity (keeps only last `capacity` bytes), zero-length data (no-op)
  - `read(): Uint8Array` ‚Äî returns a copy of current contents from oldest to newest, correctly handling wrap-around. Two cases: no wrap (slice `[0..cursor]`), wrapped (concat `[cursor..capacity] + [0..cursor]`)
  - `readString(): string` ‚Äî convenience method using `TextDecoder` for UTF-8 terminal output
  - `clear()` ‚Äî resets cursor and wrapped flag without zeroing the backing array
  - `size: number` ‚Äî current byte count (capacity when wrapped, cursor when not)
  - `maxCapacity: number` ‚Äî the fixed capacity in bytes

### Test coverage (32 tests)
- **Constructor**: specified capacity, default 1MB, zero/negative capacity throws
- **Write < capacity**: single write, multiple writes, correct size
- **Write > capacity (wrap-around)**: single large write, sequential writes causing wrap, double capacity, much larger than capacity
- **Write = capacity**: exact fit at once, exact fit with multiple writes
- **Write zero bytes**: no-op on empty buffer, no-op on existing data
- **Read empty buffer**: returns empty Uint8Array
- **Read returns a copy**: mutation of returned array doesn't affect buffer
- **Sequential operations**: complex multi-step write/read sequences, alternating small writes
- **Clear**: resets size, read returns empty, allows re-use, works after wrap-around
- **readString**: empty, ASCII, Unicode (emoji + diacritics), after wrap-around
- **Binary data**: null bytes, ANSI escape sequences
- **Capacity of 1**: edge case with single-byte buffer

## Issue #137: PTY data coalescing (5ms timer) ‚Äî COMPLETED

**Status:** Done
**Date:** 2026-03-05

### Summary
Added a 5ms coalescing timer per PTY instance in the PTY host, matching VS Code's `TerminalDataBufferer` pattern. When `pty.onData` fires, data is buffered in an array. After 5ms of quiet, all buffered chunks are joined and emitted as a single `data` event. This reduces the number of IPC messages by an order of magnitude for burst output (e.g., build logs, `cat` of large files) while adding imperceptible latency for interactive typing.

### What was done

1. **`packages/server/src/pty-host.ts`**:
   - Added `COALESCE_INTERVAL_MS` constant (5ms) and `CoalesceBuffer` interface (array of string chunks + timer handle).
   - Added `coalesceBuffers` Map tracking per-PTY coalescing state.
   - Added `flushCoalesceBuffer(id)` helper ‚Äî clears the timer, deletes the buffer entry, joins all chunks, and emits a single `data` event. Called by: the 5ms timer, `handleResize` (flush before resize), `pty.onExit` (flush before exit event), `handleKill` (flush before kill), and shutdown cleanup.
   - Added `bufferData(id, data)` helper ‚Äî if no buffer exists, creates one with a 5ms timer; if a buffer exists, pushes the chunk onto the existing array (timer already running).
   - Changed `pty.onData` callback from `emit({ type: "data", ... })` to `bufferData(cmd.id, data)`.
   - Updated `pty.onExit` to call `flushCoalesceBuffer(cmd.id)` before emitting exit event ‚Äî ensures consumers see all output before the process is marked as exited.
   - Updated `handleResize` to call `flushCoalesceBuffer(cmd.id)` before `pty.resize()` ‚Äî ensures output is associated with the correct terminal dimensions (matching VS Code's behavior per PRD-terminal-perf.md).
   - Updated `handleKill` to call `flushCoalesceBuffer(cmd.id)` before `pty.kill()` ‚Äî ensures no output is lost if kill is synchronous.

### Design decisions
- Used an array of strings (`chunks: string[]`) for the coalescing buffer rather than incremental string concatenation. This avoids O(n¬≤) copying during the coalescing window ‚Äî strings are only joined once when the timer fires. Consistent with the PRD-terminal-perf.md specification.
- The coalescing timer is per-PTY, not global. Each PTY instance has its own independent buffer and timer. This prevents one high-throughput terminal from delaying output from other terminals.
- Flush is called in multiple places (timer, resize, exit, kill, shutdown) as a belt-and-suspenders approach. Each flush is idempotent ‚Äî if the buffer doesn't exist, `flushCoalesceBuffer` returns immediately. This ensures no data is lost regardless of the event ordering.
- The `onExit` handler flushes before emitting the exit event. This is critical because node-pty may deliver final output chunks asynchronously just before the exit callback, and without flushing, those chunks would be lost (the coalescing timer wouldn't fire because the PTY is already gone from the map).
- The `handleKill` also flushes before killing. On some platforms, `pty.kill()` may be synchronous and the `onExit` handler may fire inline. Flushing in both places is safe (idempotent) and covers all edge cases.

### Design decisions
- The ack sends the *actual* accumulated character count rather than a fixed 5,000. If a single WebSocket message contains 8,000 characters, the ack will be `{"chars":8000}`. If there's a remainder (e.g., 5,500 chars ‚Üí ack 5,500, reset to 0 instead of 500), the next ack will simply be delayed by at most one `CHAR_COUNT_ACK_SIZE` worth of characters ‚Äî negligible given the 100K high watermark.
- No separate `useEffect` cleanup is needed for the flow control counter ‚Äî it's a simple `useRef` that resets on `ws.onopen` and is garbage-collected with the component.

## Issue #136: IPC buffer optimization (O(n¬≤) ‚Üí array accumulator)

**Status:** Done
**Priority:** Medium (performance optimization)

### Summary
Replaced the O(n¬≤) string concatenation pattern in both the PTY host's stdin line reader and the PtyHostClient's stdout line reader with array-based accumulators. Under high throughput, the previous `buffer += chunk` pattern copies the entire accumulated string on every concatenation, creating GC pressure. The new pattern pushes chunks onto an array and joins only when scanning for newlines, keeping the remainder as a single-element array.

### Files modified

1. **`packages/server/src/pty-host.ts`** (stdin reader, `readStdin` function):
   - Replaced `let buffer = ""` + `buffer += chunk` with `const bufferChunks: string[]` array accumulator.
   - On each chunk: push to array, join once to scan for newlines, use `searchStart` index to avoid re-scanning already-processed portions.
   - After draining all complete lines, keep the remainder (after the last newline) as a single element in `bufferChunks`.
   - On stdin close: join remaining chunks and process.

2. **`packages/server/src/services/pty-host-client.ts`** (stdout reader, `drainLines` function):
   - Replaced `let stdoutBuffer = ""` + `stdoutBuffer += chunk.toString()` with `const stdoutChunks: string[]` array accumulator.
   - `drainLines()` now takes no arguments: joins `stdoutChunks`, clears the array, scans for newlines with `searchStart` index, and keeps the remainder.
   - `on("end")` handler joins and processes any remaining data.

### Verification
- `tsc --noEmit` passes (only pre-existing informational messages)
- `ultracite check` passes on both modified files with zero issues
- **All 52 tests pass** (32 ring buffer + 14 PTY host + 6 terminal manager ‚Äî no regressions)

### Acceptance criteria status
- [x] PTY host stdin reader uses array-based accumulator instead of `buffer += chunk`
- [x] PtyHostClient stdout reader uses array-based accumulator instead of `buffer += chunk`
- [x] No behavioral change ‚Äî line-delimited JSON parsing works identically (verified by all 52 tests passing)
- [x] Existing tests pass without modification
- [x] Tests: high-throughput scenario (`seq 1 1000` in existing coalescing test) completes without issues

### Design decisions
- Both readers use the same pattern: `chunks: string[]` ‚Üí `join("")` ‚Üí scan with `indexOf("\n", searchStart)` ‚Üí `slice(searchStart)` for remainder. The `searchStart` index avoids re-scanning already-processed portions of the joined string.
- The array is cleared with `chunks.length = 0` (fastest way to clear an array in V8) before keeping the remainder as a single element.
- The join-once-scan-with-index approach is O(n) per chunk batch (one join, one scan pass) instead of O(n¬≤) from repeated slice-and-reassign.
