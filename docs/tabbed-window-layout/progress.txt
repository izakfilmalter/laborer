## Issue #3: Layout-utils: workspace tile operations ‚Äî COMPLETED

Date: 2026-03-13

### What was done
- Created new module `apps/web/src/panels/workspace-tile-utils.ts` with pure functions:
  - `addWorkspaceToTab(tab, workspaceId)` ‚Äî add a workspace as a new tile leaf; creates single leaf for empty tab, flattens into horizontal split for existing horizontal root, wraps in new horizontal split otherwise
  - `removeWorkspaceFromTab(tab, workspaceId)` ‚Äî remove a workspace tile by ID, collapses single-child splits, returns undefined layout when last workspace removed
  - `splitWorkspaceTile(tab, workspaceId, newWorkspaceId, direction)` ‚Äî split a workspace tile to add a new workspace beside it; same-direction splits are flattened (no unnecessary nesting), different-direction splits create nested split nodes
  - `resizeWorkspaceTiles(tab, nodeId, direction)` ‚Äî resize workspace tiles by walking up the tile tree to find a matching-orientation ancestor split, shifts 5% between the target and its sibling, enforces 5% minimum size
  - `reorderWorkspaceTiles(tab, workspaceOrder)` ‚Äî reorder workspace tiles based on workspace ID order; collects all leaves, reorders them, rebuilds as flat horizontal split; preserves panel tab state

  - Added `windowLayout` column: `Schema.NullOr(WindowLayoutSchema)`, nullable, default null ‚Äî stores the new hierarchical `WindowLayout` tree
  - Added `activeWindowTabId` column: nullable text ‚Äî stores the ID of the currently active window tab
  - Legacy columns (`layoutTree`, `activePaneId`, `workspaceOrder`) preserved and marked `@deprecated`

  - `windowTabSwitched` updates `activeWindowTabId`
  - `windowTabsReordered` updates tab order in the layout
  - `panelTabCreated` adds panel tabs to workspaces
  - `panelTabClosed` removes panel tabs
  - `panelTabSwitched` updates active panel tab
  - `panelTabsReordered` reorders panel tabs
  - `windowLayoutRestored` for startup/reconciliation
  - `windowLayoutSplit` with nested panel split trees
  - `windowLayoutPaneClosed` and `windowLayoutPaneAssigned`

## Issue #6: Layout migration: flat PanelNode to hierarchical WindowLayout ‚Äî COMPLETED

Date: 2026-03-13

### What was done
- Created new module `apps/web/src/panels/layout-migration.ts` with pure functions:
  - `migrateToWindowLayout(oldTree, activePaneId, workspaceOrder)` ‚Äî main migration entry point that converts a legacy flat `PanelNode` tree to the new `WindowLayout` hierarchy
  - `convertLeafNode(node)` ‚Äî converts old `LeafNode` to `PanelLeafNode`, stripping sidebar toggle flags (`diffOpen`, `devServerOpen`, `devServerTerminalId`)
  - `convertPanelTree(node)` ‚Äî recursively converts old `PanelNode` tree to `PanelTreeNode` tree (LeafNode‚ÜíPanelLeafNode, SplitNode‚ÜíPanelSplitNode)
  - `collectSidebarFlags(node)` ‚Äî aggregates sidebar toggle flags from all leaves in a workspace's sub-tree for promotion to independent panel tabs
  - `buildWorkspaceTile(workspaceId, subTree, activePaneId)` ‚Äî creates a `WorkspaceTileLeaf` for a workspace, including promoted sidebar panel tabs
  - `resolveFocusedPaneId(tree, activePaneId)` ‚Äî maps legacy `activePaneId` to the correct `focusedPaneId` per workspace
  - `sortWorkspaceIdsByOrder(workspaceIds, workspaceOrder)` ‚Äî sorts workspace IDs according to legacy `workspaceOrder` array

- Migration logic:
  1. Extracts unique workspace IDs from the old flat tree using `getWorkspaceIds()`
  2. Sorts workspaces according to `workspaceOrder` (if provided)
  3. Filters old tree per workspace using `filterTreeByWorkspace()`
  4. Converts old node types to new types (strips sidebar flags)
  5. Promotes sidebar toggle flags to independent panel tabs:
     - `diffOpen: true` ‚Üí additional `PanelTab` with `paneType: 'diff'`
     - `devServerOpen: true` ‚Üí additional `PanelTab` with `paneType: 'devServerTerminal'` (preserving `devServerTerminalId`)
  6. Wraps each workspace's panels in a `WorkspaceTileLeaf`
  7. Arranges multiple workspace tiles in a horizontal `WorkspaceTileSplit` (single workspace = direct leaf, no wrapper)
  8. Wraps everything in a single `WindowTab` inside a `WindowLayout`

### Tests
- 47 tests in `apps/web/test/layout-migration.test.ts` covering:
  - convertLeafNode: simple terminal, preserves terminalId/workspaceId/both, strips diffOpen/devServerOpen/devServerTerminalId, preserves diff/review/devServerTerminal paneTypes
  - convertPanelTree: single leaf, split with two leaves, preserves sizes, nested splits, strips sidebar flags from nested trees
  - collectSidebarFlags: no flags, diffOpen, devServerOpen with terminalId, aggregation from splits, deeply nested, diffOpen:false
  - migrateToWindowLayout ‚Äî single workspace: single-leaf tree, split tree, activePaneId‚ÜífocusedPaneId, null activePaneId fallback, cross-workspace activePaneId fallback
  - migrateToWindowLayout ‚Äî sidebar flag promotion: diffOpen‚Üídiff tab, devServerOpen‚ÜídevServer tab, both flags, false flags, main tab active, no devServerTerminalId, flags from split children
  - migrateToWindowLayout ‚Äî multi-workspace: multiple tiles in horizontal split, equal sizes, workspaceOrder respected, partial workspaceOrder, activePaneId per-workspace mapping, mixed sidebar flags
  - migrateToWindowLayout ‚Äî edge cases: no workspaceId, nested splits single workspace, deeply nested multi-workspace, empty workspaceOrder, correct tags at every level, immutability, all workspaces with diffOpen, unique IDs
  - migrateToWindowLayout ‚Äî round-trip structure: complex tree with ordering + sidebar flags + nested splits

## Issue #7: Tab bar component ‚Äî COMPLETED

Date: 2026-03-14

### What was done
- Created new presentational component `apps/web/src/components/ui/tab-bar.tsx`:
  - `TabBar` ‚Äî shared tab bar component used at both window tab level and workspace panel tab level
  - `TabBarTab` ‚Äî individual draggable tab with close button, active indicator, and dirty indicator
  - `TabBarInner` ‚Äî inner component that mounts drag-and-drop monitor only when rendered

- Component features:
  - **Auto-hide**: when `autoHide` prop is true and items.length <= 1, renders nothing
  - **Active tab indicator**: bottom border line (`bg-primary`) on the active tab
  - **Close button**: per-tab close button with `onClose(id)` callback; close click does not bubble to `onSelect`
  - **New tab (+) button**: renders a `+` icon button with configurable tooltip via `newTabTooltip` prop
  - **Dirty indicator**: amber dot with title="Unsaved changes" when `isDirty` is true
  - **Icon support**: optional `icon` ReactNode rendered before the label
  - **Drag-and-drop reordering**: using `@atlaskit/pragmatic-drag-and-drop` with per-instance `barId` to prevent cross-bar drops; visual drop indicators (left/right `bg-primary` lines); `onReorder(fromIndex, toIndex)` callback
  - **Overflow scrolling**: tab area uses `overflow-x-auto scrollbar-none` for horizontal scroll when tabs exceed width
  - **Keyboard accessibility**: `role="tablist"` on container, `role="tab"` on each item, `aria-selected` reflecting active state, `tabIndex=0` for active / `-1` for inactive, Enter/Space key handlers for selection

- Props interface:
  - `items: readonly TabBarItem[]` ‚Äî ordered tab items with `id`, `label`, `icon?`, `isDirty?`, `isActive`
  - `onSelect(id)`, `onClose(id)`, `onNew()`, `onReorder(fromIndex, toIndex)` ‚Äî callbacks
  - `autoHide?: boolean` ‚Äî hide when 0 or 1 items (defaults to false)
  - `newTabTooltip?: string` ‚Äî tooltip for the + button (defaults to "New tab")
  - `className?: string` ‚Äî additional classes for the root container

- Styling follows existing patterns: Tailwind CSS, `cn()` utility, `h-8 border-b` header bar pattern, `size-3.5` icon sizing, ghost variant buttons

### Tests

  - Rendering items: all items rendered, labels, aria-selected, dirty indicator, icon
  - Callbacks: onSelect on click, onClose on close button click, close doesn't trigger onSelect, onNew on + button, custom newTabTooltip
  - Keyboard accessibility: role=tablist, role=tab, tabIndex, Enter key, Space key

  - className prop: applied to root element
