## Issue #6: Layout migration: flat PanelNode to hierarchical WindowLayout — COMPLETED

Date: 2026-03-13

### What was done
- Created new module `apps/web/src/panels/layout-migration.ts` with pure functions:
  - `migrateToWindowLayout(oldTree, activePaneId, workspaceOrder)` — main migration entry point that converts a legacy flat `PanelNode` tree to the new `WindowLayout` hierarchy
  - `convertLeafNode(node)` — converts old `LeafNode` to `PanelLeafNode`, stripping sidebar toggle flags (`diffOpen`, `devServerOpen`, `devServerTerminalId`)
  - `convertPanelTree(node)` — recursively converts old `PanelNode` tree to `PanelTreeNode` tree (LeafNode→PanelLeafNode, SplitNode→PanelSplitNode)
  - `collectSidebarFlags(node)` — aggregates sidebar toggle flags from all leaves in a workspace's sub-tree for promotion to independent panel tabs
  - `buildWorkspaceTile(workspaceId, subTree, activePaneId)` — creates a `WorkspaceTileLeaf` for a workspace, including promoted sidebar panel tabs
  - `resolveFocusedPaneId(tree, activePaneId)` — maps legacy `activePaneId` to the correct `focusedPaneId` per workspace
  - `sortWorkspaceIdsByOrder(workspaceIds, workspaceOrder)` — sorts workspace IDs according to legacy `workspaceOrder` array

- Migration logic:
  1. Extracts unique workspace IDs from the old flat tree using `getWorkspaceIds()`
  2. Sorts workspaces according to `workspaceOrder` (if provided)
  3. Filters old tree per workspace using `filterTreeByWorkspace()`
  4. Converts old node types to new types (strips sidebar flags)
  5. Promotes sidebar toggle flags to independent panel tabs:
     - `diffOpen: true` → additional `PanelTab` with `paneType: 'diff'`
     - `devServerOpen: true` → additional `PanelTab` with `paneType: 'devServerTerminal'` (preserving `devServerTerminalId`)
  6. Wraps each workspace's panels in a `WorkspaceTileLeaf`
  7. Arranges multiple workspace tiles in a horizontal `WorkspaceTileSplit` (single workspace = direct leaf, no wrapper)
  8. Wraps everything in a single `WindowTab` inside a `WindowLayout`

### Tests
- 47 tests in `apps/web/test/layout-migration.test.ts` covering:
  - convertLeafNode: simple terminal, preserves terminalId/workspaceId/both, strips diffOpen/devServerOpen/devServerTerminalId, preserves diff/review/devServerTerminal paneTypes
  - convertPanelTree: single leaf, split with two leaves, preserves sizes, nested splits, strips sidebar flags from nested trees
  - collectSidebarFlags: no flags, diffOpen, devServerOpen with terminalId, aggregation from splits, deeply nested, diffOpen:false
  - migrateToWindowLayout — single workspace: single-leaf tree, split tree, activePaneId→focusedPaneId, null activePaneId fallback, cross-workspace activePaneId fallback
  - migrateToWindowLayout — sidebar flag promotion: diffOpen→diff tab, devServerOpen→devServer tab, both flags, false flags, main tab active, no devServerTerminalId, flags from split children
  - migrateToWindowLayout — multi-workspace: multiple tiles in horizontal split, equal sizes, workspaceOrder respected, partial workspaceOrder, activePaneId per-workspace mapping, mixed sidebar flags
  - migrateToWindowLayout — edge cases: no workspaceId, nested splits single workspace, deeply nested multi-workspace, empty workspaceOrder, correct tags at every level, immutability, all workspaces with diffOpen, unique IDs
  - migrateToWindowLayout — round-trip structure: complex tree with ordering + sidebar flags + nested splits

## Issue #22: Workspace uniqueness enforcement — COMPLETED

Date: 2026-03-14

### What was done
- Added 3 new pure functions to `apps/web/src/panels/window-tab-utils.ts` for workspace uniqueness enforcement:

- **`removeWorkspaceFromLayout(layout, workspaceId, removeFromTab)`** — removes a workspace from whatever tab it currently lives in across the entire layout. Uses `findWorkspaceLocation` to locate the workspace and the injected `removeFromTab` callback to strip it from the owning tab. Returns the layout unchanged (referential equality) if the workspace is not found.

- **`moveWorkspace(layout, workspaceId, targetTabId, removeFromTab, addToTab)`** — moves a workspace from its current location to a specific target tab. If the workspace already lives in the target tab, returns the layout unchanged (no-op). Otherwise removes it from the old tab and adds it to the target tab. Both `removeFromTab` and `addToTab` are injected as callbacks to avoid circular imports with `workspace-tile-utils.ts`.

- **`addWorkspaceToTabUnique(layout, workspaceId, targetTabId, removeFromTab, addToTab)`** — the primary entry point for within-window uniqueness enforcement. Delegates to `moveWorkspace` to ensure the workspace ends up only in the target tab. This function is used by the UI handler to enforce the constraint that a workspace can only be open in one window tab at a time.

- Updated `handleAddWorkspaceToCurrentTab` in `apps/web/src/routes/-hooks/use-panel-layout.ts`:
  - Made the handler `async` to support cross-window uniqueness checking
  - Added **cross-window uniqueness**: calls `focusExistingWindowForWorkspace(workspaceId)` before adding. If the workspace is already open in another Electron window, focuses that window instead of duplicating the workspace.
  - Added **within-window uniqueness**: calls `findWorkspaceLocation(base, workspaceId)` to check if the workspace already exists in the current layout. If already in the target tab, returns early (no-op). Otherwise calls `addWorkspaceToTabUnique` to remove the workspace from its old tab and add it to the active tab.
  - Imported `addWorkspaceToTabUnique`, `findWorkspaceLocation` from `window-tab-utils.ts` and `removeWorkspaceFromTab` from `workspace-tile-utils.ts`

### Architecture decisions
- **Dependency injection for `removeFromTab`/`addToTab`**: The pure functions in `window-tab-utils.ts` accept `removeFromTab` and `addToTab` as callback parameters rather than importing them directly from `workspace-tile-utils.ts`. This avoids circular import issues and keeps the modules decoupled — `window-tab-utils.ts` operates on `WindowLayout` while `workspace-tile-utils.ts` operates on `WindowTab`.
- **Cross-window check via Electron IPC**: The cross-window uniqueness check reuses the existing `focusExistingWindowForWorkspace` function from `desktop.ts`, which communicates with other Electron windows via the desktop bridge IPC. This is the same pattern used in `handleAssignTerminalToPane` for terminal sidebar navigation.
- **No LiveStore cross-window query**: The PRD mentions "query other windows' layouts via LiveStore" for cross-window enforcement. However, the existing `focusExistingWindowForWorkspace` Electron IPC mechanism already handles this correctly — it checks all windows' workspace assignments and focuses the right window. Using IPC is more reliable than querying LiveStore because it handles the focus-switching atomically.
