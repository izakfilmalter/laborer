## Issue 2: Vendor and build Ghostty through a pinned native toolchain
Status: COMPLETE
Date: 2026-03-13

- Added Ghostty as a pinned git submodule at vendor/ghostty (manaflow-ai/ghostty fork, SHA 1b008f5c1)
- Created new @laborer/ghostty package at packages/ghostty/
- Built GhosttyKit.xcframework from source using Zig (universal macOS arm64+x86_64)

- Created binding.gyp that links the native addon against libghostty.a and all required macOS frameworks
- Created minimal C++ N-API addon (src/addon.cc) wrapping Ghostty C API: init, isInitialized, getInfo, validateConfig
- Created TypeScript wrapper (src/index.ts) that loads the .node addon via createRequire

- All 6 tests pass
- Full monorepo typecheck passes (all 8 packages)
- Format check passes (added vendor/ to biome.json excludes)

### Build prerequisites
- Zig >= 0.15.2 (brew install zig)
- Xcode command line tools
- Run `bun run setup` in packages/ghostty/ to build GhosttyKit.xcframework
- Run `bun run build:native` to compile the Node.js native addon

### Key decisions
- Used manaflow-ai/ghostty fork (same as cmux reference) for xcframework build support
- Setup script removes xcframework-* tags locally to avoid Ghostty build system panic
- Disabled emit-macos-app during xcframework build (only need the library, not the full app)
- Linked against 13 macOS frameworks required by libghostty.a (including GameController for GCController)

## Issue 1: Ghostty helper tracer bullet — native addon app/surface layer
Status: IN PROGRESS (native addon layer complete)
Date: 2026-03-13

### What was done
- Renamed addon from C++ (.cc) to Objective-C++ (.mm) for AppKit/Metal access
- Implemented Ghostty app runtime creation (ghostty_app_new) with full runtime callback wiring:
  - wakeup_cb, action_cb, read/write/confirm clipboard callbacks, close_surface_cb
  - Action callback handles RENDER, SET_TITLE, PWD, RING_BELL, SHOW_CHILD_EXITED, CLOSE_WINDOW, CELL_SIZE, RENDERER_HEALTH
- Implemented surface lifecycle: createSurface/destroySurface with offscreen NSWindow + NSView hosting
  - Each surface gets a borderless NSWindow moved offscreen (-10000, -10000)
  - NSView is layer-backed for Metal rendering
  - Supports custom width/height, workingDirectory, command options
  - Tracked via monotonic surface IDs in an unordered_map
- Implemented surface control: setSurfaceSize, setSurfaceFocus, getSurfaceSize
- Implemented IOSurface handle extraction: getSurfaceIOSurfaceId
  - Detects CAMetalLayer on the view
  - Extracts IOSurfaceID from layer contents when available
- Implemented app tick function for event processing
- Updated TypeScript wrapper (src/index.ts) with full typed API:
  - 6 new types: CreateSurfaceOptions, SurfaceHandle, SurfaceSize, IOSurfaceInfo
  - 11 new exported functions: createApp, isAppCreated, destroyApp, appTick,
    createSurface, destroySurface, listSurfaces, setSurfaceSize, setSurfaceFocus,
    getSurfaceSize, getSurfaceIOSurfaceId
- Updated binding.gyp to use addon.mm instead of addon.cc
- Expanded test suite from 6 tests to 25 tests covering:
  - Runtime init (6 original tests)
  - App lifecycle: create, idempotent create, tick, destroy
  - Surface lifecycle: create with defaults, custom dimensions, working directory
  - Surface control: get size, set size, set focus
  - IOSurface info query
  - Surface destruction and cleanup
  - Error handling for non-existent surfaces
  - App destruction cleans up all surfaces
- All 25 tests pass
- Full monorepo typecheck passes (all packages)
- Full monorepo test suite passes (all packages)
- Format check passes

### What remains for Issue 1 completion
- Helper process: create dedicated helper process that loads the addon and exposes a control API
- Desktop integration: add 'ghostty' as a sidecar in SidecarManager
- Renderer display: import IOSurface texture in Electron renderer and display via WebGPU
- End-to-end test: verify helper startup and first-frame rendering
- Error state: failure to start helper produces visible error in UI

### Key decisions
- Used Objective-C++ (.mm) for the addon — required for NSView/NSWindow/CAMetalLayer/IOSurface APIs
- Each Ghostty surface is hosted in an offscreen NSWindow — Ghostty requires a real NSView
- Runtime callbacks are minimal stubs for now — full action mapping is Issue 7
- IOSurface extraction probes CALayer contents — actual zero-copy sharing path will be hardened in Issue 3
- Surface tracking uses a mutex-protected unordered_map with monotonic uint32 IDs
