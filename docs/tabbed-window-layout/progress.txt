## Issue #3: Layout-utils: workspace tile operations — COMPLETED

Date: 2026-03-13

### What was done
- Created new module `apps/web/src/panels/workspace-tile-utils.ts` with pure functions:
  - `addWorkspaceToTab(tab, workspaceId)` — add a workspace as a new tile leaf; creates single leaf for empty tab, flattens into horizontal split for existing horizontal root, wraps in new horizontal split otherwise
  - `removeWorkspaceFromTab(tab, workspaceId)` — remove a workspace tile by ID, collapses single-child splits, returns undefined layout when last workspace removed
  - `splitWorkspaceTile(tab, workspaceId, newWorkspaceId, direction)` — split a workspace tile to add a new workspace beside it; same-direction splits are flattened (no unnecessary nesting), different-direction splits create nested split nodes
  - `resizeWorkspaceTiles(tab, nodeId, direction)` — resize workspace tiles by walking up the tile tree to find a matching-orientation ancestor split, shifts 5% between the target and its sibling, enforces 5% minimum size
  - `reorderWorkspaceTiles(tab, workspaceOrder)` — reorder workspace tiles based on workspace ID order; collects all leaves, reorders them, rebuilds as flat horizontal split; preserves panel tab state

### Tests

  - removeWorkspaceFromTab: empty layout, not found, only workspace (undefined), two-workspace collapse, three-workspace redistribute, nested collapse, deep cascade collapse, immutability
  - splitWorkspaceTile: empty tab, not found, root leaf horizontal/vertical, same-direction flattening (H→H, V→V), different-direction nesting (V→H), nested workspace split, immutability
  - resizeWorkspaceTiles: empty tab, not found, root leaf, grow right/down, shrink left/up, minimum size enforcement, direction mismatch, edge (no sibling), nested resize, immutability
  - reorderWorkspaceTiles: empty tab, single workspace, two/three workspace reorder, unchanged order, partial order (appends unmatched), nonexistent IDs ignored, nested flatten, panel tab state preservation
  - Combined operations: add+remove round-trip, add+split+remove, add+reorder+resize

## Issue #5: Schema + LiveStore events for new layout model — COMPLETED

Date: 2026-03-13

### What was done
- Updated `panel_layout` state table in `packages/shared/src/schema.ts`:
  - Made `layoutTree` column nullable (was required) with `Schema.NullOr(PanelNodeSchema)` for backward compat
  - Added `windowLayout` column: `Schema.NullOr(WindowLayoutSchema)`, nullable, default null — stores the new hierarchical `WindowLayout` tree
  - Added `activeWindowTabId` column: nullable text — stores the ID of the currently active window tab
  - Legacy columns (`layoutTree`, `activePaneId`, `workspaceOrder`) preserved and marked `@deprecated`

- Defined shared `windowLayoutEventSchema` for all new events:
  - `{ windowId: String, windowLayout: WindowLayoutSchema, activeWindowTabId: NullOr(String) }`

- Added 12 new LiveStore events:
  - **Window tab events**: `windowTabCreated`, `windowTabClosed`, `windowTabSwitched`, `windowTabsReordered`
  - **Panel tab events**: `panelTabCreated`, `panelTabClosed`, `panelTabSwitched`, `panelTabsReordered`
  - **Hierarchical layout events**: `windowLayoutRestored`, `windowLayoutSplit`, `windowLayoutPaneClosed`, `windowLayoutPaneAssigned`

- Added materializers for all 12 new events:
  - All use the same upsert pattern: `insert({ windowId, windowLayout, activeWindowTabId }).onConflict('windowId', 'update', { windowLayout, activeWindowTabId })`
  - New events only update `windowLayout` and `activeWindowTabId` columns — they do NOT touch legacy columns
  - Legacy events continue to only update `layoutTree` and `activePaneId` — they do NOT touch new columns
  - This ensures legacy and hierarchical data can coexist during migration

- Old events (`layoutSplit`, `layoutPaneClosed`, `layoutPaneAssigned`, `layoutRestored`, `layoutWorkspacesReordered`) remain unchanged for backward compatibility with existing eventlog data

- Updated existing test assertions in `packages/shared/test/schema.test.ts` to include new columns (`windowLayout: null`, `activeWindowTabId: null`) in `deepStrictEqual` checks

- Added 15 new tests in `packages/shared/test/schema.test.ts`:
  - `windowTabCreated` materializes correctly (verifies new columns populated, legacy columns null)
  - `windowTabClosed` updates layout after closing a tab
  - `windowTabSwitched` updates `activeWindowTabId`
  - `windowTabsReordered` updates tab order in the layout
  - `panelTabCreated` adds panel tabs to workspaces
  - `panelTabClosed` removes panel tabs
  - `panelTabSwitched` updates active panel tab
  - `panelTabsReordered` reorders panel tabs
  - `windowLayoutRestored` for startup/reconciliation
  - `windowLayoutSplit` with nested panel split trees
  - `windowLayoutPaneClosed` and `windowLayoutPaneAssigned`
  - Multiple windows with hierarchical layouts stored independently
  - Hierarchical events do not overwrite legacy columns
  - Legacy events do not overwrite hierarchical columns
  - Nested workspace tile splits round-trip correctly through JSON serialization
